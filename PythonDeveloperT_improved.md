# Единый конспект Backend / Data Engineering / DLH

Этот документ собран как единый учебный конспект для подготовки к backend/data engineering/DLH-направлению. Логика построена от базового backend и API к тестированию, CI/CD, микросервисам и Data Lakehouse. Такой порядок помогает сначала понять, где рождаются данные и бизнес-логика, а затем увидеть, как эти данные проверяются, доставляются, передаются между сервисами и попадают в аналитическую платформу.

## Как пользоваться конспектом

Конспект лучше читать по порядку: сначала разобраться с HTTP API и архитектурой FastAPI-приложения, затем перейти к pytest, после этого изучить CI/CD, микросервисы и DLH. Перед собеседованием удобно повторять короткие блоки в начале каждой большой темы, схемы и итоговую шпаргалку в конце.

```text
Backend/API
    ↓
Pytest
    ↓
CI/CD
    ↓
Микросервисы
    ↓
Data Platform / DLH
    ↓
S3 + Parquet + Spark + Iceberg
    ↓
BI / Analytics
```

## Backend и API

> **Краткий ответ.** Backend — это серверная часть приложения, которая принимает HTTP-запросы, выполняет бизнес-логику, работает с базой данных и возвращает клиенту ответ. В современных системах backend обычно предоставляет API, валидирует входные данные, проверяет авторизацию и взаимодействует с другими сервисами. Для меня это базовый слой, где данные появляются в системе и откуда дальше могут уходить в микросервисы, очереди и аналитическую платформу.

### Основы HTTP и API

Когда пользователь нажимает кнопку в приложении, открывает страницу сайта или отправляет форму, клиентское приложение обращается к серверу через API.

API (Application Programming Interface) — это интерфейс взаимодействия между двумя системами. Он определяет набор правил, по которым одна система может отправлять запросы другой системе и получать ответы.

В большинстве современных приложений API строится поверх протокола HTTP.

```text
Client
   ↓
HTTP Request
   ↓
Backend
   ↓
HTTP Response
   ↓
Client
```

HTTP-запрос обычно содержит:

* метод;
* URL;
* заголовки;
* тело запроса (если требуется).

Пример запроса на создание студента:

```http
POST /students
Content-Type: application/json

{
  "name": "Ivan",
  "age": 20
}
```

После обработки запроса сервер возвращает ответ:

```http
201 Created

{
  "id": 1,
  "name": "Ivan",
  "age": 20
}
```

Сам HTTP ничего не знает про студентов, пользователей или заказы. Для него существуют только запросы и ответы. Бизнес-смысл появляется уже внутри backend-приложения.

#### HTTP-методы

На практике чаще всего используются пять методов.

**GET** используется для получения данных.

```http
GET /students/1
```

Такой запрос не должен изменять состояние системы.

**POST** используется для создания новых сущностей.

```http
POST /students
```

После успешного создания объекта обычно возвращается код `201 Created`.

**PUT** выполняет полное обновление объекта.

Если объект содержит десять полей, а в запросе переданы только пять, остальные поля считаются отсутствующими.

**PATCH** используется для частичного обновления.

```http
PATCH /students/1
```

Изменяются только переданные поля.

**DELETE** удаляет сущность.

```http
DELETE /students/1
```

#### Статус-коды

Статус-код показывает результат выполнения запроса.

Наиболее часто встречаются следующие группы:

| Код | Значение              |
| --- | --------------------- |
| 200 | OK                    |
| 201 | Created               |
| 400 | Bad Request           |
| 401 | Unauthorized          |
| 403 | Forbidden             |
| 404 | Not Found             |
| 500 | Internal Server Error |

Особенно важно понимать разницу между кодами 401 и 403.

Код 401 означает, что пользователь не прошёл аутентификацию.

Код 403 означает, что пользователь аутентифицирован, но не имеет необходимых прав.

#### Path и Query параметры

Часть данных может передаваться через URL.

Path-параметры используются для идентификации конкретного ресурса.

```http
GET /students/15
```

Здесь число `15` является идентификатором студента.

Query-параметры чаще используются для фильтрации, поиска и пагинации.

```http
GET /students?city=Moscow&page=2
```

Сервер получает дополнительные параметры:

```text
city=Moscow
page=2
```

Такой подход используется практически во всех REST API.

#### Headers и Cookies

Помимо URL и тела запроса существуют заголовки.

Через заголовки обычно передаются:

* информация об авторизации;
* тип передаваемых данных;
* технические параметры запроса.

Например:

```http
Authorization: Bearer <token>
```

Через этот заголовок обычно передаётся JWT-токен.

Cookies решают похожую задачу, но хранятся на стороне клиента и автоматически отправляются браузером при каждом запросе.

В современных backend-приложениях используются оба подхода. Выбор зависит от архитектуры системы и требований безопасности.

### Архитектурные стили API

Хотя HTTP определяет способ передачи данных, он не определяет структуру самого API. Для этого существуют различные архитектурные подходы.

На практике чаще всего используются REST, SOAP и GraphQL.

#### REST

REST (Representational State Transfer) является самым распространённым стилем построения API.

Главная идея REST заключается в том, что система представляется в виде набора ресурсов.

Например:

```text
/students
/groups
/teachers
```

Каждый ресурс поддерживает стандартные HTTP-операции.

```http
GET    /students
POST   /students
GET    /students/1
PATCH  /students/1
DELETE /students/1
```

REST хорошо сочетается с HTTP, легко документируется и отлично подходит для микросервисной архитектуры. Именно поэтому большинство современных backend-систем используют этот подход.

#### SOAP

SOAP появился значительно раньше REST и строится вокруг XML-сообщений.

Упрощённо взаимодействие выглядит следующим образом:

```xml
<Envelope>
    <Body>
        ...
    </Body>
</Envelope>
```

SOAP обеспечивает строгую спецификацию взаимодействия между системами и предоставляет встроенные механизмы безопасности.

Сегодня SOAP чаще всего встречается в банковских системах, государственных сервисах и корпоративных интеграциях.

#### GraphQL

GraphQL предлагает другой подход к взаимодействию между клиентом и сервером.

Вместо множества endpoint используется единая точка входа.

Клиент самостоятельно определяет, какие данные ему нужны.

Пример запроса:

```graphql
{
  student(id: 1) {
    name
    age
  }
}
```

Сервер вернёт только те поля, которые были явно запрошены.

GraphQL особенно удобен для сложных клиентских приложений, где разные экраны требуют разные наборы данных.

Однако реализация GraphQL обычно сложнее, чем реализация классического REST API.

### FastAPI

FastAPI — современный Python-фреймворк для создания HTTP API. Он появился значительно позже Django и Flask, но быстро стал популярным благодаря высокой производительности, удобной работе с типизацией и встроенной поддержке асинхронности.

Одной из главных особенностей FastAPI является активное использование type hints. Благодаря этому фреймворк автоматически выполняет валидацию данных, строит документацию и помогает разработчику находить ошибки ещё на этапе написания кода.

Минимальное приложение выглядит следующим образом:

```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/")
async def root():
    return {"message": "Hello World"}
```

После запуска приложения FastAPI автоматически создаёт HTTP endpoint, который начинает принимать запросы.

#### Endpoint

Endpoint — это конкретная точка входа в API.

Например:

```http
GET /students
POST /students
GET /students/1
```

Каждый endpoint обычно отвечает за одну конкретную операцию.

В FastAPI endpoint описывается через декоратор:

```python
@app.get("/students")
async def get_students():
    return []
```

Декоратор сообщает фреймворку, какой URL должен вызывать данную функцию.

#### Router

По мере роста приложения количество endpoint начинает быстро увеличиваться.

Если хранить все маршруты в одном файле, проект становится трудно поддерживать.

Поэтому маршруты обычно группируются по доменам.

Например:

```text
routers/

students.py
groups.py
teachers.py
```

Внутри каждого файла создаётся собственный APIRouter:

```python
from fastapi import APIRouter

router = APIRouter(
    prefix="/students",
    tags=["Students"]
)
```

После этого маршруты регистрируются в основном приложении.

Такой подход позволяет разделять функциональность и делает код более читаемым.

#### Swagger и OpenAPI

Одной из сильнейших сторон FastAPI является автоматическая генерация документации.

После запуска приложения обычно становятся доступны две страницы:

```text
/docs
/redoc
```

Документация строится на основе стандарта OpenAPI.

Разработчику не нужно вручную описывать endpoint, параметры запросов и структуры ответов. FastAPI делает это автоматически на основе кода приложения.

Swagger особенно полезен для:

* тестирования API;
* знакомства с новым сервисом;
* интеграции между командами;
* написания автотестов.

В реальных проектах Swagger часто становится первой точкой знакомства с новым сервисом.

#### Асинхронность

Большинство современных backend-приложений выполняет большое количество операций ввода-вывода:

* обращения к базе данных;
* вызовы внешних API;
* работу с файловыми хранилищами;
* взаимодействие с очередями сообщений.

Во время ожидания ответа процессор практически ничего не делает.

Асинхронность позволяет использовать это время более эффективно.

В FastAPI асинхронный endpoint выглядит следующим образом:

```python
@app.get("/students")
async def get_students():
    ...
```

Ключевое слово `async` сообщает Python, что функция может приостанавливать своё выполнение во время ожидания внешних операций.

Важно понимать, что асинхронность не делает вычисления быстрее. Она помогает эффективнее использовать ресурсы сервера в ситуациях, когда приложение много ожидает ответы от других систем.

#### Depends и Dependency Injection

В реальном приложении endpoint редко работает самостоятельно.

Чаще всего ему необходимы дополнительные зависимости:

* подключение к базе данных;
* сервис авторизации;
* объект текущего пользователя;
* конфигурация приложения.

Передавать такие объекты вручную неудобно.

Для решения этой задачи FastAPI использует механизм Dependency Injection через функцию `Depends`.

Пример получения подключения к базе данных:

```python
from fastapi import Depends

@app.get("/students")
async def get_students(
    db = Depends(get_db)
):
    ...
```

В данном случае FastAPI автоматически вызывает функцию `get_db()` и передаёт результат в endpoint.

Благодаря этому код становится более модульным и удобным для тестирования.

На практике через Depends часто передаются:

```python
db = Depends(get_db)

current_user = Depends(get_current_user)

settings = Depends(get_settings)
```

Фактически Depends является одним из ключевых инструментов построения архитектуры FastAPI-приложений.

### Аутентификация и авторизация

При работе с API необходимо понимать две разные задачи.

Первая задача — определить, кто именно выполняет запрос.

Вторая задача — определить, что этому пользователю разрешено делать.

Эти задачи называются аутентификацией и авторизацией.

Аутентификация отвечает на вопрос:

> Кто ты?

Авторизация отвечает на вопрос:

> Что тебе разрешено?

Например, пользователь успешно вошёл в систему.

Это означает, что аутентификация выполнена.

Однако если этот пользователь пытается удалить чужие данные и получает отказ, то сработала авторизация.

Несмотря на тесную связь, это разные механизмы.

#### JWT

Наиболее популярным способом аутентификации современных API является JWT.

JWT (JSON Web Token) представляет собой специальный токен, содержащий информацию о пользователе.

Упрощённо JWT состоит из трёх частей:

```text
Header
Payload
Signature
```

Payload обычно содержит:

```json
{
  "user_id": 1,
  "role": "admin"
}
```

После успешного входа в систему сервер выдаёт токен клиенту.

При последующих запросах клиент отправляет его через заголовок:

```http
Authorization: Bearer <token>
```

Сервер проверяет подпись токена и определяет пользователя.

Главное преимущество JWT заключается в том, что серверу не обязательно хранить информацию о сессии.

Вся необходимая информация уже содержится внутри токена.

#### Access Token и Refresh Token

На практике обычно используются два разных токена.

Access Token имеет небольшой срок жизни.

Например:

```text
15 минут
30 минут
1 час
```

Именно он используется при выполнении запросов к API.

Refresh Token живёт значительно дольше.

Например:

```text
7 дней
30 дней
90 дней
```

Его задача — получать новый Access Token без повторного ввода логина и пароля.

Такая схема позволяет повысить безопасность системы.

Если злоумышленник получит Access Token, срок его действия будет ограничен.

#### Cookies

Альтернативным способом хранения информации о пользователе являются cookies.

После входа в систему сервер может сохранить специальный cookie в браузере пользователя.

При последующих запросах браузер будет автоматически отправлять его обратно.

JWT и cookies не являются взаимоисключающими подходами.

В реальных проектах они часто используются совместно.

Например:

* Access Token хранится в памяти приложения;
* Refresh Token хранится в HttpOnly Cookie.

Такой подход считается достаточно безопасным и широко применяется в современных веб-приложениях.

### Работа с данными

После того как запрос попадает в backend-приложение, возникает следующая задача — сохранить или получить данные.

В современных Python-проектах обычно используются несколько уровней абстракции:

```text
HTTP Request
     ↓
Pydantic
     ↓
Service
     ↓
DAO
     ↓
SQLAlchemy
     ↓
Database
```

Каждый слой отвечает за собственную задачу и позволяет избежать смешивания бизнес-логики, валидации и работы с базой данных.

#### Pydantic

Pydantic отвечает за валидацию и преобразование входящих данных.

Представим запрос:

```json
{
  "name": "Ivan",
  "age": 20
}
```

Перед тем как использовать эти данные, необходимо убедиться, что они имеют корректную структуру.

Для этого создаётся схема:

```python
from pydantic import BaseModel

class StudentCreate(BaseModel):
    name: str
    age: int
```

Теперь FastAPI автоматически проверит входящий запрос.

Если клиент отправит:

```json
{
  "name": "Ivan",
  "age": "twenty"
}
```

то запрос завершится ошибкой ещё до попадания в бизнес-логику.

Это позволяет существенно сократить количество проверок внутри приложения.

Pydantic используется не только для входящих запросов, но и для ответов сервера.

Например:

```python
class StudentResponse(BaseModel):
    id: int
    name: str
    age: int
```

В результате API начинает работать через явно описанные контракты данных.

#### SQLAlchemy

Если Pydantic отвечает за структуру данных в API, то SQLAlchemy отвечает за взаимодействие с базой данных.

SQLAlchemy является ORM (Object Relational Mapper).

Главная идея ORM заключается в том, чтобы работать не напрямую с SQL-запросами, а с Python-объектами.

Например, таблица студентов:

```sql
CREATE TABLE students (
    id SERIAL PRIMARY KEY,
    name TEXT,
    age INTEGER
);
```

может быть представлена в виде модели:

```python
from sqlalchemy.orm import Mapped
from sqlalchemy.orm import mapped_column

class Student(Base):
    __tablename__ = "students"

    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str]
    age: Mapped[int]
```

Теперь каждая строка таблицы становится объектом Python.

Вместо написания SQL вручную можно работать через ORM-модели.

Например:

```python
student = Student(
    name="Ivan",
    age=20
)
```

SQLAlchemy самостоятельно сформирует необходимый SQL-запрос.

Это не означает, что SQL становится ненужным. Хороший backend-разработчик всё равно должен понимать, какие запросы выполняются внутри ORM.

Однако ORM позволяет писать более читаемый и поддерживаемый код.

#### Pydantic и SQLAlchemy

Новички часто путают эти сущности.

Важно понимать различие.

Pydantic работает на уровне API.

Он отвечает за проверку и преобразование данных.

SQLAlchemy работает на уровне базы данных.

Он отвечает за сохранение и получение данных.

Упрощённо:

```text
Pydantic
↓
API Layer

SQLAlchemy
↓
Database Layer
```

В большинстве FastAPI-проектов используются обе технологии одновременно.

#### DAO

По мере роста проекта возникает новая проблема.

Если SQLAlchemy-запросы начинают появляться прямо внутри endpoint, код быстро превращается в трудно поддерживаемый набор логики.

Например:

```python
@app.post("/students")
async def create_student(...):
    student = Student(...)
    session.add(student)
    session.commit()
```

Для небольшого проекта это допустимо.

Для production-системы — уже нет.

Поэтому появляется отдельный слой доступа к данным.

Он называется DAO (Data Access Object).

DAO отвечает исключительно за работу с базой данных.

Например:

```python
class StudentDAO:

    @classmethod
    async def create(
        cls,
        session,
        student_data
    ):
        ...
```

Теперь endpoint больше не знает деталей работы с SQLAlchemy.

Он просто вызывает DAO.

Такой подход даёт несколько преимуществ:

* упрощает тестирование;
* уменьшает связанность компонентов;
* делает код более читаемым;
* позволяет переиспользовать запросы.

В большинстве backend-проектов DAO становится границей между бизнес-логикой и базой данных.

#### Service Layer

Хотя в небольших приложениях можно работать напрямую через DAO, в большинстве production-систем появляется дополнительный уровень — сервисный слой.

Сервис отвечает за бизнес-логику.

Например:

```python
class StudentService:

    async def create_student(...):
        ...
```

Именно здесь могут находиться:

* проверки бизнес-правил;
* работа с несколькими DAO;
* вызовы внешних сервисов;
* публикация событий в Kafka;
* отправка уведомлений.

Получается следующая структура:

```text
Router
  ↓
Service
  ↓
DAO
  ↓
Database
```

Такое разделение считается одной из наиболее распространённых архитектур backend-приложений.

#### Alembic

Со временем структура базы данных начинает меняться.

Появляются новые таблицы.

Добавляются новые поля.

Изменяются индексы.

Если выполнять такие изменения вручную, рано или поздно возникнут проблемы с синхронизацией окружений.

Для решения этой задачи используется Alembic.

Alembic является системой миграций для SQLAlchemy.

Миграция представляет собой описание изменения схемы базы данных.

Например:

```python
def upgrade():
    op.add_column(
        "students",
        sa.Column(
            "email",
            sa.String()
        )
    )
```

После применения миграции структура базы изменяется автоматически.

На практике чаще всего используются две команды.

Создание миграции:

```bash
alembic revision --autogenerate -m "add email"
```

Применение миграций:

```bash
alembic upgrade head
```

Таким образом схема базы данных начинает храниться в репозитории вместе с кодом приложения.

Это особенно важно при командной разработке.

### Инфраструктура

Даже идеально написанное приложение не имеет особой ценности, если его невозможно запустить на другом компьютере или сервере.

Для решения этой проблемы используется контейнеризация.

#### Docker

Docker позволяет упаковать приложение вместе со всеми зависимостями в контейнер.

Главная идея заключается в том, что приложение должно запускаться одинаково:

* на ноутбуке разработчика;
* на тестовом стенде;
* на production-сервере.

Обычно проект содержит Dockerfile.

Например:

```dockerfile
FROM python:3.12

WORKDIR /app

COPY . .

RUN pip install -r requirements.txt

CMD ["uvicorn", "main:app"]
```

Docker собирает образ приложения и позволяет запускать его в любом совместимом окружении.

Благодаря этому исчезает классическая проблема:

> У меня на компьютере всё работает.

#### Docker Compose

На практике приложение редко состоит только из FastAPI.

Обычно одновременно используются:

* PostgreSQL;
* Redis;
* Kafka;
* MinIO;
* другие сервисы.

Запускать каждый контейнер вручную неудобно.

Для этого используется Docker Compose.

Пример конфигурации:

```yaml
services:

  app:
    build: .

  postgres:
    image: postgres:16
```

После запуска одной команды:

```bash
docker compose up
```

поднимается всё окружение проекта.

Это делает локальную разработку значительно проще.

На практике Docker Compose часто используется как стандартный способ запуска проекта для новых разработчиков.

### Практический проект

После изучения отдельных технологий важно понять, как они работают вместе внутри реального приложения.

В качестве примера рассмотрим сервис управления студентами, реализованный на FastAPI.

Проект позволяет создавать, получать, изменять и удалять студентов через HTTP API. Несмотря на простоту предметной области, такая архитектура практически полностью повторяет структуру production-сервисов.

#### Архитектура проекта

На уровне компонентов приложение выглядит следующим образом:

```text
Client
   ↓
Router
   ↓
Service
   ↓
DAO
   ↓
SQLAlchemy
   ↓
PostgreSQL
```

Каждый слой имеет собственную зону ответственности.

Router отвечает за взаимодействие с HTTP.

Service содержит бизнес-логику.

DAO работает с базой данных.

SQLAlchemy формирует SQL-запросы.

PostgreSQL хранит данные.

Главная цель такого разделения — уменьшить связанность компонентов и упростить поддержку приложения.

#### Структура проекта

Упрощённо проект может выглядеть следующим образом:

```text
app/

├── main.py
├── database.py
├── models/
├── schemas/
├── dao/
├── services/
├── routers/
└── migrations/
```

Каждая директория отвечает за отдельную часть системы.

В результате разработчик всегда понимает, где искать нужную логику.

#### Модель данных

Начнём с SQLAlchemy-модели.

```python
class Student(Base):
    __tablename__ = "students"

    id: Mapped[int] = mapped_column(
        primary_key=True
    )

    name: Mapped[str]

    age: Mapped[int]
```

Эта модель соответствует таблице в PostgreSQL.

Каждая запись в таблице становится объектом класса Student.

#### Pydantic-схемы

Для работы с API используются отдельные схемы.

Схема создания студента:

```python
class StudentCreate(BaseModel):
    name: str
    age: int
```

Схема ответа:

```python
class StudentResponse(BaseModel):
    id: int
    name: str
    age: int
```

Важно понимать, что SQLAlchemy-модели и Pydantic-схемы решают разные задачи.

Модель описывает структуру таблицы.

Схема описывает структуру API.

Хотя поля могут выглядеть одинаково, это разные уровни приложения.

#### DAO в проекте

Слой DAO инкапсулирует работу с базой данных.

Например:

```python
class StudentDAO:

    @classmethod
    async def create(
        cls,
        session,
        student_data
    ):
        student = Student(**student_data)

        session.add(student)

        await session.commit()

        return student
```

Теперь остальная часть приложения не знает деталей работы с SQLAlchemy.

Она просто вызывает методы DAO.

#### Service Layer в проекте

Сервисный слой содержит бизнес-логику.

Например:

```python
class StudentService:

    @staticmethod
    async def create_student(
        session,
        data
    ):
        return await StudentDAO.create(
            session,
            data
        )
```

На первый взгляд такой слой кажется лишним.

Однако со временем именно здесь появляются:

* проверки бизнес-правил;
* работа с несколькими DAO;
* интеграции с другими сервисами;
* отправка событий;
* аудит действий пользователя.

Поэтому в production-проектах сервисный слой практически всегда присутствует.

#### Router в проекте

Router становится входной точкой приложения.

Например:

```python
@router.post(
    "/students",
    response_model=StudentResponse
)
async def create_student(
    student: StudentCreate,
    session=Depends(get_db)
):
    return await StudentService.create_student(
        session,
        student.model_dump()
    )
```

Здесь происходит несколько важных вещей.

FastAPI получает HTTP-запрос.

Pydantic валидирует данные.

Depends создаёт подключение к базе.

Сервисный слой выполняет бизнес-логику.

После этого результат возвращается клиенту.

#### Полный путь запроса POST /students

Теперь можно проследить жизненный цикл одного запроса от начала до конца.

Пусть клиент отправляет следующий запрос:

```http
POST /students

{
  "name": "Ivan",
  "age": 20
}
```

Сначала запрос попадает в Router.

FastAPI определяет нужный endpoint и создаёт объект StudentCreate.

Если структура данных не соответствует схеме, выполнение останавливается и клиент получает ошибку валидации.

Если данные корректны, управление передаётся в Service.

Сервис выполняет бизнес-логику и вызывает DAO.

DAO создаёт SQLAlchemy-модель и сохраняет её через PostgreSQL.

После успешного сохранения объект возвращается обратно через все уровни приложения.

В итоге клиент получает ответ:

```json
{
  "id": 1,
  "name": "Ivan",
  "age": 20
}
```

Таким образом один запрос проходит через несколько независимых слоёв, каждый из которых отвечает за собственную задачу.

#### Где используется Depends

Во время изучения FastAPI может показаться, что Depends нужен только для базы данных.

На практике его применение гораздо шире.

Например, через Depends часто передаются:

```python
db = Depends(get_db)

user = Depends(get_current_user)

settings = Depends(get_settings)
```

Таким образом endpoint не создаёт зависимости самостоятельно.

FastAPI делает это автоматически.

Подобный подход упрощает тестирование и делает код более модульным.

#### Где появляются тесты

Даже небольшой сервис обычно покрывается несколькими уровнями тестов.

Сначала тестируется бизнес-логика.

Например:

```python
StudentService.create_student()
```

Затем тестируются DAO и взаимодействие с базой данных.

После этого появляются API-тесты.

Например:

```http
POST /students
```

В результате можно отдельно проверять каждый уровень системы и быстро находить источник ошибки.

Именно здесь позже появится Pytest, который станет основным инструментом автоматизированного тестирования.

#### Как такая архитектура развивается дальше

На первых этапах приложение может существовать как единый сервис.

Однако со временем система начинает расти.

Появляются новые домены:

```text
Students

Groups

Payments

Notifications
```

Каждая область может быть выделена в отдельный сервис.

Тогда вместо одной базы данных и одного приложения появляется набор взаимодействующих микросервисов.

В этот момент начинают использоваться:

* Kafka;
* API Gateway;
* Docker;
* Kubernetes;
* CI/CD.

Данные, которые создаются через API, начинают использоваться не только самим сервисом, но и аналитическими системами.

Например, после создания студента сервис может отправить событие:

```text
StudentCreated
```

Это событие может попасть в Kafka, затем сохраниться в S3, обработаться Spark и в конечном итоге оказаться в Iceberg-таблице аналитической платформы.

Именно поэтому понимание базового backend-приложения является фундаментом для изучения микросервисов, Data Engineering и DLH.

На этом уровне уже можно увидеть полный путь данных — от HTTP-запроса пользователя до записи в базу данных. В следующих главах этот путь будет расширяться и постепенно превратится в полноценную распределённую систему.

Перед переходом к тестированию важно зафиксировать связь: backend-код ценен не только тогда, когда он написан, но и тогда, когда его поведение можно стабильно проверять. Поэтому следующая большая тема — pytest как основной инструмент автоматической проверки Python/FastAPI/ETL-логики.

## Pytest

> **Краткий ответ.** Pytest — это фреймворк для автоматизированного тестирования Python-кода. Он позволяет писать простые unit-тесты, готовить окружение через fixtures, проверять разные сценарии через parametrize и расширять поведение тестового запуска через хуки и плагины. В backend и data-проектах pytest помогает проверять бизнес-логику, API, работу с БД и ETL-компоненты, а затем запускать эти проверки в CI/CD.

### Основы Pytest

Тестирование является неотъемлемой частью современной разработки. Чем сложнее становится система, тем выше вероятность появления ошибок после внесения изменений в код.

Ручная проверка быстро перестаёт справляться с этой задачей. Именно поэтому появляются автоматизированные тесты.

Pytest позволяет запускать код в контролируемых условиях и автоматически проверять ожидаемый результат.

Простейший тест выглядит следующим образом:

```python
def test_sum():
    assert 2 + 2 == 4
```

Во время запуска pytest выполняет функцию и проверяет выражение после ключевого слова `assert`.

Если условие истинно, тест считается успешным.

Если условие ложно, тест завершается ошибкой.

Например:

```python
def test_sum():
    assert 2 + 2 == 5
```

В этом случае pytest покажет подробную информацию о причине падения.

Одним из преимуществ pytest является его минимализм. Для написания простого теста не требуется наследование от специальных классов или использование сложных конструкций.

Достаточно обычной Python-функции.

### Test Discovery

Прежде чем запустить тесты, pytest должен их найти.

Этот процесс называется Test Discovery.

По умолчанию pytest ищет файлы по определённым шаблонам:

```text
test_*.py

*_test.py
```

Например:

```text
tests/

test_students.py

test_auth.py

student_test.py
```

Все такие файлы будут автоматически найдены во время запуска.

Аналогичные правила действуют и для функций.

По умолчанию pytest считает тестами функции, имя которых начинается с префикса:

```python
def test_create_student():
    ...
```

Функция:

```python
def create_student():
    ...
```

тестом считаться уже не будет.

Именно поэтому важно соблюдать принятые соглашения об именовании.

### Запуск тестов

Наиболее простой вариант запуска выглядит так:

```bash
pytest
```

В этом случае pytest рекурсивно просканирует проект и выполнит все найденные тесты.

Запуск конкретного файла:

```bash
pytest tests/test_students.py
```

Запуск конкретного теста:

```bash
pytest tests/test_students.py::test_create_student
```

Очень часто используются дополнительные флаги.

Подробный вывод:

```bash
pytest -v
```

Вывод print-сообщений:

```bash
pytest -s
```

Остановка после первой ошибки:

```bash
pytest -x
```

На практике именно комбинация:

```bash
pytest -v
```

используется чаще всего.

### Результаты выполнения тестов

После завершения запуска pytest показывает итоговую статистику.

Например:

```text
10 passed
```

означает, что все тесты успешно завершились.

Также могут встречаться следующие статусы:

```text
PASSED
FAILED
SKIPPED
XFAILED
XPASSED
ERROR
```

Важно различать FAILED и ERROR.

FAILED означает, что тест выполнился, но проверка оказалась ложной.

Например:

```python
assert 2 == 3
```

ERROR означает, что ошибка произошла ещё до выполнения проверки.

Например:

```python
1 / 0
```

или проблема при создании тестового окружения.

На собеседованиях этот вопрос задают достаточно часто.

### Структура тестового проекта

По мере роста количества тестов появляется необходимость организовывать их структуру.

Чаще всего используется следующая схема:

```text
project/

├── app/
├── tests/
│
├── test_api.py
├── test_service.py
├── conftest.py
│
└── pytest.ini
```

Директория `tests` содержит сами тесты.

Файл `conftest.py` содержит общие фикстуры.

Файл `pytest.ini` используется для конфигурации pytest.

Такой подход считается стандартом для большинства Python-проектов.

### Конфигурация через pytest.ini

Pytest позволяет централизованно настраивать поведение через специальный конфигурационный файл.

Например:

```ini
[pytest]

testpaths = tests

python_files = test_*.py

markers =
    slow
    smoke
```

Через конфигурацию можно:

* изменять правила поиска тестов;
* регистрировать маркеры;
* задавать дополнительные параметры запуска;
* подключать плагины.

В крупных проектах pytest.ini практически всегда присутствует.

### Почему Pytest стал стандартом

До появления pytest в Python активно использовался встроенный модуль unittest.

Однако pytest предложил более удобный подход.

Вместо:

```python
self.assertEqual(
    result,
    expected
)
```

можно писать:

```python
assert result == expected
```

Код становится значительно короче и читается гораздо легче.

Кроме того pytest предоставляет:

* фикстуры;
* параметризацию;
* хуки;
* плагины;
* удобную интеграцию с CI/CD.

Именно поэтому сегодня большинство современных Python-команд использует pytest как основной инструмент тестирования.

### Fixtures

По мере роста количества тестов быстро возникает проблема подготовки тестового окружения.

Представим, что для каждого теста необходимо:

* создать подключение к базе данных;
* подготовить тестового пользователя;
* создать HTTP-клиент;
* заполнить тестовые данные.

Если писать этот код внутри каждого теста, проект быстро превратится в набор дублирующихся конструкций.

Для решения этой задачи в pytest используются fixtures.

Фикстура представляет собой функцию, которая создаёт и возвращает некоторое тестовое окружение.

Простейший пример:

```python
import pytest

@pytest.fixture
def student():
    return {
        "name": "Ivan",
        "age": 20
    }
```

Теперь эту фикстуру можно использовать в тесте:

```python
def test_student_name(student):
    assert student["name"] == "Ivan"
```

Pytest автоматически найдёт фикстуру, вызовет её и передаст результат в тест.

Это одна из ключевых возможностей pytest.

Фактически fixtures являются механизмом Dependency Injection для тестов.

Очень часто на собеседованиях можно провести аналогию:

```text
FastAPI
↓
Depends()

Pytest
↓
Fixture
```

Обе конструкции решают похожую задачу — предоставляют зависимость потребителю.

#### Setup и Teardown

Во многих случаях недостаточно просто создать объект.

После завершения теста необходимо выполнить очистку.

Например:

* закрыть подключение к базе;
* удалить временный файл;
* остановить тестовый сервис.

Такой процесс обычно называют:

```text
Setup
↓
Test
↓
Teardown
```

Именно здесь появляется механизм `yield`.

#### Yield Fixtures

Фикстура с yield состоит из двух частей.

Всё, что находится до yield, относится к setup.

Всё, что находится после yield, относится к teardown.

Пример:

```python
import pytest

@pytest.fixture
def db_connection():

    connection = create_connection()

    yield connection

    connection.close()
```

Последовательность выполнения выглядит следующим образом:

```text
Create Connection
        ↓
      Yield
        ↓
      Test
        ↓
Close Connection
```

Это один из наиболее распространённых паттернов в pytest.

Практически все интеграционные тесты используют yield-фикстуры.

Например:

* работа с PostgreSQL;
* работа с Kafka;
* работа с Redis;
* работа с Docker-контейнерами.

#### Scope

По умолчанию фикстура создаётся для каждого теста отдельно.

Такое поведение называется:

```python
scope="function"
```

Это самый безопасный вариант.

Однако иногда создание объекта занимает много времени.

Например:

```text
PostgreSQL Container

Spark Session

Kafka Cluster
```

Создавать такие объекты перед каждым тестом слишком дорого.

Для решения этой задачи используются области видимости.

На практике чаще всего встречаются следующие варианты.

##### Function Scope

Создаётся перед каждым тестом.

```python
@pytest.fixture(
    scope="function"
)
```

Это значение по умолчанию.

Каждый тест получает полностью независимое окружение.

##### Class Scope

Создаётся один раз для тестового класса.

```python
@pytest.fixture(
    scope="class"
)
```

Используется значительно реже.

##### Module Scope

Создаётся один раз на модуль.

```python
@pytest.fixture(
    scope="module"
)
```

Все тесты внутри файла используют один экземпляр фикстуры.

##### Session Scope

Создаётся один раз за весь запуск pytest.

```python
@pytest.fixture(
    scope="session"
)
```

Часто используется для:

* тестовых контейнеров;
* подключения к тестовой БД;
* запуска локального Kafka-кластера.

Важно помнить, что увеличение scope повышает скорость выполнения тестов, но одновременно увеличивает риск влияния тестов друг на друга.

Поэтому использовать session scope нужно осторожно.

#### Fixture Dependencies

Фикстуры могут использовать другие фикстуры.

Например:

```python
@pytest.fixture
def db():
    ...

@pytest.fixture
def user(db):
    ...
```

Сначала pytest создаст db.

После этого создаст user.

Получается своеобразный граф зависимостей.

Именно поэтому fixtures хорошо подходят для сложных тестовых окружений.

Например:

```text
Database
    ↓
Repository
    ↓
Service
    ↓
API Client
```

Pytest автоматически построит правильный порядок создания объектов.

#### Autouse Fixtures

Иногда требуется выполнять фикстуру автоматически.

Например:

* очистка базы данных;
* настройка переменных окружения;
* инициализация логирования.

Для этого используется параметр autouse.

Пример:

```python
@pytest.fixture(
    autouse=True
)
def setup_environment():
    ...
```

Теперь фикстура будет запускаться для каждого теста автоматически.

Явно передавать её больше не требуется.

Использовать autouse следует аккуратно.

Если таких фикстур становится слишком много, тестовое окружение начинает выглядеть магическим и становится трудно понять, что именно выполняется перед тестом.

#### Params

Иногда одна и та же фикстура должна возвращать разные значения.

Для этого используется параметр params.

Например:

```python
@pytest.fixture(
    params=[
        "postgres",
        "sqlite"
    ]
)
def database(request):
    return request.param
```

Теперь каждый тест будет выполнен дважды.

Первый раз:

```text
postgres
```

Второй раз:

```text
sqlite
```

Это удобно для проверки поведения системы в разных конфигурациях.

#### Name

По умолчанию имя фикстуры совпадает с именем функции.

Однако иногда это поведение нужно изменить.

Например:

```python
@pytest.fixture(
    name="db"
)
def database_connection():
    ...
```

Теперь в тестах будет использоваться имя:

```python
db
```

хотя сама функция называется иначе.

Такая возможность помогает делать API фикстур более понятным.

### conftest.py

По мере роста проекта количество фикстур начинает увеличиваться.

Если хранить их рядом с тестами, очень быстро возникает хаос.

Для решения этой проблемы используется специальный файл:

```text
conftest.py
```

Это один из важнейших файлов любого pytest-проекта.

#### Зачем нужен conftest.py

Представим структуру проекта:

```text
tests/

├── test_students.py
├── test_groups.py
└── test_auth.py
```

Во всех тестах используется одна и та же фикстура базы данных.

Вместо копирования её в каждый файл достаточно поместить её в:

```text
tests/conftest.py
```

После этого pytest автоматически найдёт фикстуру и сделает её доступной во всех тестах.

Никакие импорты писать не нужно.

Это важное отличие conftest.py от обычных Python-модулей.

#### Пример

```python
import pytest

@pytest.fixture
def db():
    return Database()
```

После этого любой тест внутри директории может использовать:

```python
def test_create_student(db):
    ...
```

Несмотря на отсутствие import.

Pytest самостоятельно найдёт нужную фикстуру.

#### Иерархия conftest.py

В крупных проектах может существовать несколько conftest.py.

Например:

```text
tests/

├── conftest.py
│
├── api/
│   ├── conftest.py
│   └── test_api.py
│
└── db/
    ├── conftest.py
    └── test_db.py
```

Фикстуры наследуются по иерархии директорий.

Это позволяет удобно разделять тестовое окружение между различными типами тестов.

#### Практика использования

В большинстве backend-проектов в conftest.py обычно располагаются:

* тестовый клиент FastAPI;
* подключение к базе данных;
* тестовые пользователи;
* авторизационные токены;
* фабрики тестовых данных;
* общие настройки окружения.

Фактически conftest.py становится центральной точкой управления тестовой инфраструктурой проекта.

Именно поэтому хорошее понимание fixtures и conftest.py считается базовым требованием для любого Python-разработчика, работающего с pytest.

### Parametrize и Markers

Когда тестов становится много, часто возникает ситуация, когда одна и та же логика должна проверяться на разных входных данных.

Например, endpoint создания студента нужно проверить сразу в нескольких сценариях:

* корректный запрос;
* пустое имя;
* некорректный возраст;
* отсутствующее обязательное поле.

Можно написать отдельный тест на каждый сценарий, но тогда код быстро начнёт повторяться.

Для таких случаев используется параметризация.

#### Parametrize

`pytest.mark.parametrize` позволяет запускать один и тот же тест несколько раз с разными наборами данных.

Пример:

```python
import pytest

@pytest.mark.parametrize(
    "value, expected",
    [
        (1, 2),
        (2, 4),
        (3, 6),
    ]
)
def test_double(value, expected):
    assert value * 2 == expected
```

Pytest выполнит этот тест три раза.

Каждый запуск получит собственные значения `value` и `expected`.

На практике параметризация особенно полезна в API-тестах.

```python
@pytest.mark.parametrize(
    "payload, expected_status",
    [
        (
            {"name": "Ivan", "age": 20},
            201
        ),
        (
            {"name": "", "age": 20},
            422
        ),
        (
            {"name": "Ivan", "age": "wrong"},
            422
        ),
    ]
)
def test_create_student(
    client,
    payload,
    expected_status
):
    response = client.post(
        "/students",
        json=payload
    )

    assert response.status_code == expected_status
```

Такой тест проверяет несколько сценариев, но сохраняет одну общую структуру.

Параметризацию стоит использовать тогда, когда меняются только входные данные и ожидаемый результат.

Если сценарии сильно отличаются по логике, лучше писать отдельные тесты.

#### Markers

Markers позволяют помечать тесты дополнительной информацией.

Например:

```python
import pytest

@pytest.mark.smoke
def test_login():
    ...
```

После этого можно запускать только smoke-тесты:

```bash
pytest -m smoke
```

Или наоборот исключить их:

```bash
pytest -m "not smoke"
```

Markers активно используются в CI/CD.

Например, быстрые smoke-тесты можно запускать при каждом merge request, а долгие regression-тесты — по расписанию ночью.

#### Встроенные маркеры

Pytest содержит несколько встроенных маркеров.

`skip` полностью пропускает тест.

```python
@pytest.mark.skip(
    reason="Временно отключено"
)
def test_old_feature():
    ...
```

`skipif` пропускает тест только при выполнении условия.

```python
import sys
import pytest

@pytest.mark.skipif(
    sys.platform == "win32",
    reason="Не запускается на Windows"
)
def test_linux_only():
    ...
```

`xfail` используется для теста, который ожидаемо падает.

```python
@pytest.mark.xfail(
    reason="Известный баг"
)
def test_known_bug():
    assert 1 == 2
```

Если такой тест падает, pytest не считает это неожиданной ошибкой.

Это удобно, когда баг известен, но тест уже хочется сохранить в кодовой базе.

#### Пользовательские маркеры

В реальных проектах часто используются собственные маркеры.

Например:

```python
@pytest.mark.db
def test_create_student_in_database():
    ...
```

или:

```python
@pytest.mark.integration
def test_external_service():
    ...
```

Чтобы pytest не предупреждал о неизвестных маркерах, их обычно регистрируют в `pytest.ini`.

```ini
[pytest]

markers =
    smoke: быстрые критичные тесты
    db: тесты с базой данных
    integration: интеграционные тесты
```

После этого маркеры становятся частью тестовой конфигурации проекта.

### Жизненный цикл Pytest

Pytest не просто запускает все функции `test_*`.

Перед выполнением тестов он проходит несколько этапов.

Понимание этого жизненного цикла особенно важно при работе с хуками и собственными плагинами.

Общая схема выглядит так:

```text
Initialization
      ↓
Collection
      ↓
Setup
      ↓
Call
      ↓
Teardown
      ↓
Reporting
```

На этапе initialization pytest читает конфигурацию проекта, загружает плагины и файлы `conftest.py`.

На этапе collection он ищет тестовые файлы и собирает тестовые функции.

На этапе setup подготавливаются фикстуры.

На этапе call выполняется тело теста.

На этапе teardown выполняется очистка ресурсов.

На этапе reporting формируется итоговый отчёт.

#### Collection

Collection — один из самых важных этапов для понимания pytest.

На этом этапе тесты ещё не выполняются.

Pytest только формирует список тестов, которые будут запущены.

Каждый найденный тест внутри pytest представлен объектом `item`.

Именно с этими объектами работают многие хуки.

Например, можно изменить порядок тестов, добавить маркер или пропустить часть тестов до начала выполнения.

#### Setup, Call и Teardown

Каждый тест фактически состоит из трёх фаз.

Setup — подготовка окружения.

Call — выполнение тела теста.

Teardown — очистка после теста.

Если ошибка возникает в assert, обычно это фаза call.

Если ошибка возникает при создании фикстуры, это чаще всего setup.

Если ошибка возникает при очистке ресурсов, это teardown.

Такое разделение особенно важно при анализе отчётов и написании плагинов.

### Хуки Pytest

Хуки позволяют вмешиваться в работу pytest на разных этапах жизненного цикла.

Если фикстуры управляют окружением тестов, то хуки управляют поведением самого pytest.

Через хуки можно:

* регистрировать маркеры;
* менять список собранных тестов;
* получать результат выполнения теста;
* добавлять собственный вывод в отчёт;
* интегрировать pytest с внешними системами.

#### pytest_configure

`pytest_configure` вызывается после загрузки конфигурации pytest.

Часто этот хук используется для регистрации маркеров.

```python
def pytest_configure(config):
    config.addinivalue_line(
        "markers",
        "db: тесты с базой данных"
    )
```

После этого можно использовать маркер:

```python
@pytest.mark.db
def test_database():
    ...
```

Pytest больше не будет считать его неизвестным.

#### pytest_collection_modifyitems

`pytest_collection_modifyitems` вызывается после сбора всех тестов.

Он получает список `items`.

Например, можно отсортировать тесты по имени:

```python
def pytest_collection_modifyitems(
    config,
    items
):
    items.sort(
        key=lambda item: item.name
    )
```

В реальных проектах этот хук часто используется для более сложной логики.

Например, можно пропускать тесты в зависимости от версии продукта, окружения или набора маркеров.

#### pytest_runtest_makereport

`pytest_runtest_makereport` позволяет получить результат выполнения конкретного теста.

Это один из самых полезных хуков при написании плагинов.

```python
import pytest

@pytest.hookimpl(
    hookwrapper=True
)
def pytest_runtest_makereport(
    item,
    call
):
    outcome = yield

    report = outcome.get_result()

    if report.when == "call":
        item.test_result = report.outcome
```

Параметр `hookwrapper=True` позволяет выполнить код после стандартной обработки pytest.

Через `report.outcome` можно получить результат теста:

```text
passed
failed
skipped
```

Такой хук часто используется для сохранения результатов тестов во внешние системы, например в базу данных или отчётный сервис.

#### pytest_terminal_summary

`pytest_terminal_summary` вызывается в конце тестового запуска.

Через него можно добавить собственный блок в итоговый вывод pytest.

```python
def pytest_terminal_summary(
    terminalreporter,
    exitstatus
):
    terminalreporter.write_sep(
        "=",
        "Custom summary"
    )

    terminalreporter.write_line(
        "Дополнительная статистика"
    )
```

Этот хук полезен, если плагин должен показать дополнительную информацию после завершения всех тестов.

### Декораторы и pytest.hookimpl

Многие механизмы pytest основаны на декораторах.

Например:

```python
@pytest.fixture
def db():
    ...
```

или:

```python
@pytest.mark.parametrize(...)
```

Декоратор позволяет изменить поведение функции или добавить к ней дополнительную информацию.

При работе с хуками часто используется специальный декоратор:

```python
@pytest.hookimpl
```

Он позволяет управлять поведением hook-функции.

Например:

```python
@pytest.hookimpl(
    tryfirst=True
)
def pytest_configure(config):
    ...
```

`tryfirst=True` говорит pytest выполнить этот хук раньше других реализаций.

`trylast=True` делает обратное.

`hookwrapper=True` позволяет обернуть выполнение хука и получить результат после стандартной обработки.

Эти параметры особенно полезны при разработке собственных плагинов.

### Плагины Pytest

Плагин pytest — это расширение, которое добавляет новое поведение в тестовый фреймворк.

Обычные тесты проверяют код приложения. Фикстуры подготавливают окружение. Хуки позволяют вмешиваться в жизненный цикл pytest. Плагины объединяют эти возможности и позволяют переиспользовать их между проектами.

Например, плагин может добавлять новые CLI-опции, регистрировать собственные маркеры, изменять список тестов перед запуском, собирать статистику, сохранять результаты тестов во внешнюю систему или формировать собственный отчёт.

В реальных проектах часто используются готовые плагины:

* `pytest-cov`
* `pytest-xdist`
* `pytest-mock`
* `pytest-asyncio`

Но иногда команда пишет собственный плагин под внутренние процессы.

#### conftest.py и отдельный плагин

Небольшую логику можно хранить прямо в `conftest.py`. Это удобно, если она нужна только в одном проекте.

Если функциональность должна использоваться в нескольких репозиториях, её лучше вынести в отдельный pytest-плагин. Такой плагин можно установить как обычный Python-пакет и подключать в разных проектах.

### pytester

Тестировать обычные функции достаточно просто.

```python
def add(a, b):
    return a + b
```

Для такой функции достаточно обычного теста:

```python
def test_add():
    assert add(2, 3) == 5
```

Но pytest-плагин работает иначе. Он влияет не на одну функцию, а на сам запуск pytest. Например, может изменять collection, добавлять маркеры, сохранять результаты тестов или менять итоговый отчёт.

Для тестирования таких сценариев используется специальная фикстура `pytester`.

Она позволяет создать временный тестовый проект, запустить внутри него pytest и проверить результат запуска.

```python
def test_simple_pytest_run(pytester):
    pytester.makepyfile(
        """
        def test_ok():
            assert True
        """
    )

    result = pytester.runpytest()

    result.assert_outcomes(
        passed=1
    )
```

### Кейс: Version Markers

Представим продукт, у которого есть несколько версий.

```text
2.1
2.2
2.3
```

Некоторые тесты должны запускаться только начиная с определённой версии.

Например, новая функциональность появилась только в версии `2.3`.

Тест может выглядеть следующим образом:

```python
@pytest.mark.min_version("2.3")
def test_new_feature():
    ...
```

Если текущая версия продукта ниже `2.3`, такой тест необходимо автоматически пропустить.

Для решения этой задачи можно написать собственный pytest-плагин.

#### Идея плагина

Плагин должен:

1. Зарегистрировать новый маркер.
2. Во время collection пройтись по тестам.
3. Сравнить требуемую версию с текущей.
4. Добавить `skip`, если версия не подходит.

Упрощённая реализация:

```python
import pytest
from packaging.version import Version

CURRENT_VERSION = Version("2.2")


def pytest_configure(config):
    config.addinivalue_line(
        "markers",
        "min_version(version): минимальная версия продукта"
    )


def pytest_collection_modifyitems(
    config,
    items
):
    for item in items:
        marker = item.get_closest_marker(
            "min_version"
        )

        if marker is None:
            continue

        required_version = Version(
            marker.args[0]
        )

        if CURRENT_VERSION < required_version:
            item.add_marker(
                pytest.mark.skip(
                    reason="Product version is too low"
                )
            )
```

Такой плагин работает на этапе collection. Тесты ещё не выполняются, но pytest уже знает список тестов и их маркеры.

#### Тестирование плагина через pytester

```python
def test_min_version_marker_skips_test(
    pytester
):
    pytester.makepyfile(
        """
        import pytest

        @pytest.mark.min_version("2.3")
        def test_new_feature():
            assert True
        """
    )

    result = pytester.runpytest()

    result.assert_outcomes(
        skipped=1
    )
```

Этот тест проверяет не отдельную функцию, а поведение pytest при запуске временного проекта.

### Кейс: Mongo Saver

Второй пример — плагин для сохранения результатов тестов во внешнее хранилище.

Представим, что команда хочет сохранять результаты каждого тестового запуска в MongoDB.

Это может понадобиться для:

* внутренней отчётности;
* анализа стабильности тестов;
* построения дашбордов;
* хранения истории запусков.

После выполнения теста необходимо получить его имя и статус.

Например:

```json
{
  "test_name": "test_create_student",
  "status": "passed"
}
```

Для этого подходит хук `pytest_runtest_makereport`.

#### Получение результата теста

```python
import pytest

results = []


@pytest.hookimpl(
    hookwrapper=True
)
def pytest_runtest_makereport(
    item,
    call
):
    outcome = yield

    report = outcome.get_result()

    if report.when != "call":
        return

    results.append(
        {
            "test_name": item.name,
            "status": report.outcome,
        }
    )
```

Здесь важно использовать фазу `call`, поскольку она соответствует выполнению тела теста.

После завершения запуска результаты можно сохранить во внешнюю систему:

```python
def pytest_terminal_summary(
    terminalreporter,
    exitstatus
):
    save_results_to_mongo(results)
```

Такой подход разделяет сбор результатов и их сохранение.

### Monkeypatch

При тестировании плагина Mongo Saver не стоит подключаться к настоящей MongoDB.

Иначе тесты начнут зависеть от внешней инфраструктуры.

Если база данных окажется недоступна, тест упадёт не из-за ошибки в коде плагина, а из-за проблем окружения.

Для изоляции используется `monkeypatch`.

Он позволяет временно подменять функции, методы, переменные окружения и объекты.

Пример:

```python
def test_save_results(
    monkeypatch
):
    saved = []

    def fake_save_results(data):
        saved.extend(data)

    monkeypatch.setattr(
        "plugin.save_results_to_mongo",
        fake_save_results
    )
```

Теперь вместо реальной записи в MongoDB данные будут сохраняться в обычный список.

#### Тестирование Mongo Saver

```python
def test_mongo_saver_plugin(
    pytester,
    monkeypatch
):
    saved = []

    def fake_save(data):
        saved.extend(data)

    monkeypatch.setattr(
        "plugin.save_results_to_mongo",
        fake_save
    )

    pytester.makepyfile(
        """
        def test_ok():
            assert True
        """
    )

    result = pytester.runpytest()

    result.assert_outcomes(
        passed=1
    )

    assert saved[0]["status"] == "passed"
```

В этом тесте создаётся временный проект, запускается pytest, плагин получает результат выполнения теста, а сохранение в MongoDB заменяется на запись в обычный список.

Такой подход позволяет тестировать плагины полностью изолированно от внешних систем.

### Как Pytest связан с Backend

В backend-проектах pytest обычно используется на нескольких уровнях.

На уровне unit-тестов проверяется бизнес-логика:

```python
StudentService.create_student()
```

На уровне integration-тестов проверяется взаимодействие с базой данных:

```text
Service
  ↓
DAO
  ↓
PostgreSQL
```

На уровне API-тестов проверяется внешний контракт сервиса:

```http
POST /students
```

Таким образом pytest становится связующим звеном между бизнес-логикой, базой данных, API и CI/CD. Именно эти тесты позже будут автоматически запускаться в pipeline после каждого push или merge request.

После того как тесты написаны, их нужно запускать не вручную от случая к случаю, а автоматически при изменениях в коде. Поэтому pytest естественно переходит в тему CI/CD.

## CI/CD

> **Краткий ответ.** CI/CD — это автоматизация проверки, сборки и доставки изменений. В GitLab CI процесс описывается в `.gitlab-ci.yml`, где задаются pipeline, stages, jobs, переменные, артефакты и правила запуска. Для backend/data-команды CI/CD важен тем, что каждый push или merge request автоматически проходит тесты, сборку и другие проверки, поэтому ошибки находятся раньше, а релизы становятся стабильнее.

### Зачем нужен CI/CD

Представим проект без CI/CD.

Разработчик вносит изменения в код, запускает тесты локально и отправляет изменения в репозиторий.

После этого другой разработчик вручную проверяет код, запускает тесты на своём компьютере и переносит новую версию приложения на сервер.

Такой процесс работает только для очень маленьких проектов.

По мере роста команды начинают появляться проблемы:

* забыли запустить тесты;
* забыли собрать проект;
* забыли обновить конфигурацию;
* изменения работают локально, но не работают на сервере;
* несколько разработчиков одновременно ломают общий стенд.

CI/CD решает эти проблемы за счёт автоматизации.

Каждое изменение проходит через заранее описанный процесс проверки.

Разработчик больше не должен вручную выполнять десятки однотипных действий.

### Continuous Integration

Continuous Integration (CI) переводится как непрерывная интеграция.

Главная идея заключается в том, что изменения должны регулярно объединяться в основную ветку проекта и автоматически проверяться.

Типичный процесс выглядит следующим образом:

```text
Developer
    ↓
Git Push
    ↓
Pipeline
    ↓
Tests
    ↓
Result
```

После каждого push система автоматически запускает набор проверок.

Например:

* линтеры;
* unit-тесты;
* integration-тесты;
* проверки безопасности;
* сборку проекта.

Если хотя бы одна проверка завершилась ошибкой, изменения считаются неготовыми.

Таким образом проблемы обнаруживаются максимально рано.

### Continuous Delivery и Continuous Deployment

После успешного прохождения CI возникает вопрос доставки изменений.

Здесь появляются два похожих термина.

#### Continuous Delivery

В случае Continuous Delivery приложение автоматически готовится к выпуску.

Pipeline выполняет:

* сборку;
* тестирование;
* подготовку артефактов.

Но финальное решение о деплое принимает человек.

Схема выглядит так:

```text
Build
  ↓
Tests
  ↓
Ready for Deploy
  ↓
Manual Approval
  ↓
Production
```

#### Continuous Deployment

В случае Continuous Deployment дополнительного подтверждения нет.

Если все проверки успешно завершились, новая версия автоматически попадает в production.

```text
Build
  ↓
Tests
  ↓
Deploy
  ↓
Production
```

Такой подход используется реже и требует высокого уровня автоматизированного тестирования.

### Pipeline

Pipeline представляет собой последовательность шагов, которые выполняются после определённого события.

Например:

* push;
* merge request;
* расписание;
* создание тега.

Типичный pipeline может выглядеть следующим образом:

```text
Lint
  ↓
Tests
  ↓
Build
  ↓
Deploy
```

Каждый шаг запускается автоматически.

Если один из шагов завершился ошибкой, выполнение pipeline останавливается.

По сути pipeline становится формализованным описанием процесса поставки программного обеспечения.

### GitLab CI

В экосистеме GitLab описание pipeline хранится в специальном файле:

```text
.gitlab-ci.yml
```

Именно этот файл определяет:

* какие job существуют;
* в каком порядке они запускаются;
* какие образы используются;
* какие переменные доступны;
* какие артефакты сохраняются.

Простейший пример:

```yaml
stages:
  - test

tests:
  stage: test

  script:
    - pytest
```

После каждого push GitLab создаёт pipeline и запускает job `tests`.

Внутри job выполняется команда:

```bash
pytest
```

Если тесты завершились успешно, job получает статус Success.

Если тесты упали — статус Failed.

### Stages и Jobs

Pipeline состоит из stages.

Stage — это логический этап выполнения.

Например:

```yaml
stages:
  - lint
  - test
  - build
  - deploy
```

Внутри каждого stage находятся jobs.

Пример:

```yaml
lint:
  stage: lint

tests:
  stage: test

build_image:
  stage: build
```

Важно понимать разницу.

Stage определяет этап.

Job определяет конкретную задачу.

Например:

```text
Stage: Test

├── Unit Tests
├── Integration Tests
└── API Tests
```

Все три job относятся к одному этапу тестирования.

По умолчанию следующий stage запускается только после успешного завершения предыдущего.

### Runner

Сам GitLab не выполняет код.

Для запуска job используются GitLab Runner.

Runner представляет собой процесс, который получает задания от GitLab и выполняет их.

Упрощённо схема выглядит так:

```text
GitLab
   ↓
Runner
   ↓
Docker Container
   ↓
Job
```

Во многих проектах каждая job запускается внутри отдельного Docker-контейнера.

Это позволяет обеспечить воспроизводимость окружения.

Независимо от того, кто сделал commit, job будет выполняться одинаково.

### Image

Для выполнения job обычно используется Docker-образ.

Например:

```yaml
image: python:3.12
```

Теперь каждая job запускается внутри контейнера Python 3.12.

При необходимости можно использовать собственные образы компании.

Например:

```yaml
image: registry.company.ru/backend/python-ci:latest
```

Это позволяет заранее установить все необходимые зависимости.

### Script

Основная логика job описывается в секции script.

Например:

```yaml
tests:
  stage: test

  script:
    - pip install -r requirements.txt
    - pytest
```

Команды выполняются последовательно сверху вниз.

Если одна из команд завершится ошибкой, job будет считаться неуспешной.

В большинстве backend-проектов именно через script запускаются:

* тесты;
* линтеры;
* сборка Docker-образов;
* миграции базы данных.

### Первая реальная схема CI

На практике backend-проект обычно проходит следующий путь:

```text
Developer
    ↓
Git Push
    ↓
GitLab Pipeline
    ↓
Pytest
    ↓
Build Docker Image
    ↓
Deploy
```

Именно поэтому предыдущая глава про pytest напрямую связана с CI/CD.

Тесты становятся частью автоматического процесса проверки изменений.

### Artifacts

Во время выполнения pipeline часто возникает необходимость передать файлы между job.

Например, на этапе сборки может формироваться отчёт о тестировании или собираться Docker-образ. На следующем этапе эти результаты нужно использовать повторно.

Для решения этой задачи используются artifacts.

Артефактом считается любой файл или директория, которые GitLab сохраняет после завершения job.

Пример:

```yaml
tests:
  stage: test

  script:
    - pytest --junitxml=report.xml

  artifacts:
    paths:
      - report.xml
```

После завершения job файл `report.xml` будет сохранён GitLab.

Другие job смогут использовать этот файл без повторного запуска тестов.

Артефакты часто используются для хранения:

* отчётов тестирования;
* результатов сборки;
* логов;
* coverage-отчётов;
* подготовленных пакетов приложения.

При работе с pytest чаще всего сохраняются:

```yaml
artifacts:
  paths:
    - reports/
```

или:

```yaml
artifacts:
  reports:
    junit: report.xml
```

В этом случае GitLab сможет отображать результаты тестирования непосредственно в интерфейсе pipeline.

### Needs

По умолчанию GitLab выполняет stages последовательно.

Например:

```text
Lint
  ↓
Test
  ↓
Build
  ↓
Deploy
```

Следующий stage начнёт работу только после полного завершения предыдущего.

Для небольших проектов это нормально.

Однако в крупных проектах подобная схема может значительно увеличивать время выполнения pipeline.

Для решения этой проблемы используется механизм `needs`.

Пример:

```yaml
build:
  stage: build

  needs:
    - tests
```

Теперь job может запускаться сразу после завершения необходимых зависимостей, не ожидая завершения всего stage.

Упрощённо:

Без needs:

```text
Stage A
   ↓
Stage B
   ↓
Stage C
```

С needs:

```text
Tests
   ↓
Build
```

Такой подход позволяет запускать независимые задачи параллельно и существенно ускорять pipeline.

В крупных проектах использование `needs` считается одной из основных техник оптимизации CI.

### Services

Иногда тестам недостаточно только контейнера с приложением.

Например, integration-тестам может понадобиться:

* PostgreSQL;
* Redis;
* Kafka;
* MongoDB.

Для запуска дополнительных контейнеров используется секция `services`.

Пример:

```yaml
tests:
  image: python:3.12

  services:
    - postgres:16

  script:
    - pytest
```

Во время выполнения job GitLab автоматически поднимет контейнер PostgreSQL и сделает его доступным внутри тестового окружения.

Таким образом можно запускать полноценные integration-тесты прямо внутри CI.

Очень часто встречается комбинация:

```yaml
services:
  - postgres
  - redis
```

Особенно в backend-проектах.

Это позволяет тестировать взаимодействие приложения с инфраструктурой практически так же, как в production.

### Variables

Практически любой pipeline использует переменные окружения.

Например:

* пароли;
* токены;
* адреса сервисов;
* настройки окружения.

Для этого используются variables.

Пример:

```yaml
variables:
  APP_ENV: test
```

Использование:

```yaml
script:
  - echo $APP_ENV
```

Помимо переменных внутри `.gitlab-ci.yml`, GitLab позволяет хранить их в настройках проекта.

Такие переменные обычно используются для:

```text
DATABASE_PASSWORD

JWT_SECRET

DOCKER_TOKEN

AWS_ACCESS_KEY_ID
```

Чувствительные данные никогда не должны храниться в репозитории.

Для них используются Protected и Masked Variables.

Это один из самых частых вопросов на собеседованиях по GitLab CI.

### Schedule Pipeline

Не все pipeline запускаются после push.

Иногда необходимо выполнять задачи по расписанию.

Например:

* ночной запуск regression-тестов;
* обновление данных;
* генерация отчётов;
* резервное копирование.

Для этого используется Schedule Pipeline.

Упрощённая схема:

```text
00:00
  ↓
Pipeline
  ↓
Regression Tests
```

Настраивается через интерфейс GitLab.

После настройки pipeline будет запускаться автоматически в указанное время.

На практике ночные запуски используются очень часто, поскольку позволяют выполнять тяжёлые проверки без влияния на скорость разработки.

### Rules

Не каждая job должна запускаться всегда.

Например:

* deploy нужен только для main;
* regression-тесты нужны только ночью;
* build нужен только для merge request.

Для управления этим используется секция `rules`.

Пример:

```yaml
deploy:
  script:
    - deploy.sh

  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'
```

Теперь job будет запускаться только в ветке main.

Другой пример:

```yaml
rules:
  - if: '$CI_PIPELINE_SOURCE == "schedule"'
```

Такая job будет запускаться только в pipeline, созданном по расписанию.

Сегодня `rules` является основным механизмом управления запуском job.

Старые конструкции вроде `only` и `except` постепенно используются всё реже.

### Regex в GitLab CI

В правилах часто используются регулярные выражения.

Например, необходимо запускать job только для release-веток.

```text
release-1.0
release-2.0
release-3.1
```

Тогда можно использовать regex.

Пример:

```yaml
rules:
  - if: '$CI_COMMIT_BRANCH =~ /^release-.*$/'
```

GitLab проверит название ветки и выполнит job только при совпадении с шаблоном.

Ещё один популярный пример:

```yaml
rules:
  - if: '$CI_COMMIT_TAG =~ /^v\d+\.\d+\.\d+$/'
```

Подходит для тегов вида:

```text
v1.0.0
v2.3.1
v10.5.4
```

Регулярные выражения позволяют гибко управлять запуском pipeline и часто встречаются в production-конфигурациях.

### Отладка Pipeline

Даже хорошо написанный pipeline периодически ломается.

Поэтому важно понимать, как искать причину ошибки.

Первое место, куда следует смотреть, — логи job.

Каждая команда из script отображается в интерфейсе GitLab.

Например:

```yaml
script:
  - pytest
```

Если тесты упали, причина будет видна прямо в логах.

Вторая важная вещь — переменные окружения.

Очень часто ошибка связана с тем, что переменная не была передана или имеет неправильное значение.

Полезная команда:

```yaml
script:
  - env
```

Она выводит доступные переменные окружения.

Третья частая причина ошибок — артефакты.

Например:

```text
File not found
```

может означать, что предыдущая job не сохранила нужный файл.

При отладке pipeline полезно задавать себе три вопроса:

1. Выполнилась ли предыдущая job?
2. Существует ли нужный файл?
3. Доступна ли нужная переменная?

Во многих случаях этого достаточно для поиска проблемы.

### Практический пример Backend Pipeline

Представим FastAPI-сервис со следующими проверками:

```text
Lint
↓
Pytest
↓
Build Docker Image
↓
Deploy
```

Конфигурация может выглядеть следующим образом:

```yaml
stages:
  - lint
  - test
  - build

lint:
  stage: lint

  script:
    - ruff check .

tests:
  stage: test

  script:
    - pytest

build:
  stage: build

  script:
    - docker build -t app .
```

После каждого push GitLab автоматически:

1. Проверит стиль кода.
2. Запустит тесты.
3. Соберёт Docker-образ.

Таким образом изменения проходят одинаковый путь независимо от того, кто именно сделал commit.

Именно поэтому CI/CD становится связующим звеном между разработкой, тестированием и эксплуатацией системы.

В следующей главе эта идея будет расширена до уровня микросервисов, где один pipeline начинает обслуживать уже не один сервис, а целую распределённую систему.

Когда один backend-сервис начинает расти, появляется потребность разделять систему на независимые части. Поэтому после CI/CD логично перейти к микросервисам: там автоматизация, тесты и контракт API становятся уже не удобством, а необходимым условием стабильной работы.

## Микросервисы

> **Краткий ответ.** Микросервисы — это подход, при котором большая система разделяется на небольшие независимые сервисы, каждый из которых отвечает за свою бизнес-область и часто владеет собственной базой данных. Сервисы взаимодействуют через HTTP API, API Gateway или события в Kafka/RabbitMQ. Такой подход помогает масштабировать систему и команды, но требует хороших контрактов, CI/CD, мониторинга, логирования и продуманного тестирования.

### От монолита к микросервисам

Практически любая большая система начинается с монолита.

Монолит представляет собой единое приложение, внутри которого находятся все основные компоненты системы:

* API;
* бизнес-логика;
* работа с базой данных;
* фоновые задачи;
* интеграции.

Упрощённая схема выглядит следующим образом:

```text
Client
   ↓
Monolith
   ↓
Database
```

Для небольших проектов такой подход вполне оправдан.

Монолит проще разрабатывать, тестировать и разворачивать.

Однако по мере роста системы начинают появляться проблемы.

Количество разработчиков увеличивается.

Количество функций увеличивается.

Кодовая база начинает быстро расти.

В результате любое изменение начинает затрагивать всё приложение.

Именно в этот момент многие компании начинают переходить к микросервисной архитектуре.

### Что такое микросервис

Микросервис — это самостоятельное приложение, отвечающее за одну бизнес-область.

Например, вместо одного большого приложения могут существовать отдельные сервисы:

```text
Users Service

Orders Service

Payments Service

Notifications Service
```

Каждый сервис развивается независимо.

Каждый сервис может иметь собственную команду разработки.

Каждый сервис может выпускать новые версии без необходимости пересобирать всю систему.

Упрощённая схема выглядит следующим образом:

```text
Client
   ↓
API Gateway
   ↓
 ┌─────────────┬─────────────┬─────────────┐
 │             │             │
Users      Orders      Payments
Service    Service     Service
 │             │             │
 DB            DB            DB
```

Главная идея заключается в разделении ответственности между сервисами.

### Принцип единственной ответственности

Один из важнейших принципов микросервисной архитектуры — Single Responsibility Principle.

Каждый сервис должен отвечать только за одну бизнес-задачу.

Например:

```text
Users Service
↓
Пользователи
```

```text
Payments Service
↓
Платежи
```

```text
Notifications Service
↓
Уведомления
```

Если сервис начинает решать слишком много задач одновременно, он постепенно превращается в новый монолит.

Поэтому при проектировании микросервисов важно постоянно задавать вопрос:

> За какую бизнес-область отвечает данный сервис?

### Слабая связанность

Микросервисы должны быть максимально независимыми друг от друга.

Это свойство называется слабой связанностью (Loose Coupling).

Плохо:

```text
Service A
 ↓
Service B
 ↓
Service C
 ↓
Service D
```

Падение одного сервиса способно вызвать цепочку проблем во всей системе.

Хорошо:

```text
Service A

Service B

Service C

Service D
```

Каждый сервис знает только о необходимых интерфейсах взаимодействия.

Изменение внутренней реализации одного сервиса не должно ломать остальные.

Именно поэтому в микросервисах большое внимание уделяется контрактам API.

### Отдельная база данных на сервис

Одна из фундаментальных практик микросервисной архитектуры заключается в том, что каждый сервис владеет своими данными.

Нежелательная схема:

```text
Service A
      ↓
      DB
      ↑
Service B
```

В этом случае оба сервиса напрямую зависят от одной структуры данных.

Любое изменение схемы может повлиять сразу на несколько компонентов системы.

Предпочтительный вариант:

```text
Service A
    ↓
   DB A

Service B
    ↓
   DB B
```

Теперь каждый сервис самостоятельно управляет собственной схемой данных.

Это повышает независимость компонентов и упрощает развитие системы.

На практике допускаются различные варианты хранения:

* PostgreSQL;
* MySQL;
* MongoDB;
* Redis;
* S3;
* Iceberg.

Главное — данные принадлежат владельцу сервиса.

### API Gateway

Когда количество сервисов начинает расти, появляется новая проблема.

Клиенту приходится знать адрес каждого сервиса отдельно.

Например:

```text
users.company.ru

orders.company.ru

payments.company.ru
```

Такой подход быстро становится неудобным.

Для решения этой задачи используется API Gateway.

Он становится единой точкой входа в систему.

Схема выглядит следующим образом:

```text
Client
   ↓
API Gateway
   ↓
 ┌─────────────┬─────────────┬─────────────┐
 │             │             │
Users      Orders      Payments
```

Gateway может выполнять несколько задач:

* маршрутизация запросов;
* аутентификация;
* авторизация;
* rate limiting;
* логирование;
* агрегация ответов.

На практике API Gateway используется практически во всех крупных микросервисных системах.

### Синхронное взаимодействие

Самый простой способ взаимодействия сервисов — синхронный вызов.

Например:

```text
Orders Service
      ↓ HTTP
Users Service
```

Сервис отправляет запрос и ждёт ответа.

Чаще всего используются:

* HTTP REST;
* gRPC.

Пример:

```text
Orders Service
      ↓
GET /users/123
      ↓
Users Service
```

Преимущество такого подхода заключается в простоте реализации.

Однако появляется сильная зависимость между сервисами.

Если Users Service недоступен, Orders Service может перестать работать.

Поэтому для сложных систем активно используются асинхронные механизмы взаимодействия.

### Асинхронное взаимодействие

Когда сервисов становится много, синхронное взаимодействие начинает создавать проблемы.

Представим следующую ситуацию:

```text
Orders Service
      ↓
Users Service
      ↓
Payments Service
      ↓
Notifications Service
```

Каждый сервис ожидает ответ следующего.

Если один из сервисов становится недоступен, вся цепочка начинает работать нестабильно.

Кроме того, увеличивается общее время обработки запроса.

Для решения этой проблемы часто используется асинхронное взаимодействие.

Главная идея заключается в том, что сервис не ждёт немедленного ответа от другого сервиса.

Вместо этого он публикует событие и продолжает работу.

Например:

```text
Order Created
```

После публикации события другие сервисы могут обработать его самостоятельно.

Такой подход лежит в основе Event Driven Architecture.

### Event Driven Architecture

Event Driven Architecture (EDA) — архитектурный подход, основанный на обмене событиями.

Вместо прямого вызова:

```text
Orders Service
      ↓
Notifications Service
```

используется событие:

```text
Orders Service
      ↓
Order Created
      ↓
Broker
      ↓
Notifications Service
```

Сервису заказов больше не важно, кто именно будет обрабатывать событие.

Главное — сообщить системе, что произошло определённое действие.

Это позволяет уменьшить связанность между компонентами.

В крупных распределённых системах событийная архитектура используется очень часто.

Особенно в Data Platform и DLH.

### Kafka

Apache Kafka — одна из самых популярных платформ для потоковой передачи событий.

Очень часто Kafka называют брокером сообщений, хотя фактически она ближе к распределённому журналу событий.

Основная идея выглядит следующим образом:

```text
Producer
    ↓
  Topic
    ↓
Consumer
```

Producer публикует событие.

Kafka сохраняет его внутри topic.

Consumer читает событие и выполняет обработку.

Например:

```text
Student Service
      ↓
StudentCreated
      ↓
Kafka
      ↓
Analytics Service
```

Теперь аналитическая система может получать события независимо от backend-сервиса.

Это одна из причин, почему Kafka так часто используется в Data Engineering.

#### Topic

Topic можно представить как логическую категорию событий.

Например:

```text
students

orders

payments

notifications
```

Каждый topic содержит поток сообщений.

Producer записывает сообщения в topic.

Consumer читает сообщения из topic.

#### Partition

Для масштабирования Kafka использует partitions.

Например:

```text
students

├── Partition 0
├── Partition 1
└── Partition 2
```

Partition позволяет распределять нагрузку между несколькими consumer.

Именно благодаря partition Kafka способна обрабатывать огромные объёмы данных.

#### Consumer Group

Несколько consumer могут объединяться в группу.

Например:

```text
Consumer Group

├── Consumer A
├── Consumer B
└── Consumer C
```

Kafka автоматически распределяет partitions между участниками группы.

Это позволяет горизонтально масштабировать обработку событий.

### RabbitMQ

RabbitMQ решает похожую задачу, но использует другую модель.

Упрощённая схема выглядит следующим образом:

```text
Producer
    ↓
 Queue
    ↓
Consumer
```

Сообщение помещается в очередь.

После обработки сообщение обычно удаляется.

В Kafka сообщение сохраняется в журнале событий.

В RabbitMQ сообщение чаще всего существует до момента обработки.

Из-за этого области применения отличаются.

RabbitMQ хорошо подходит для:

* фоновых задач;
* очередей уведомлений;
* обработки команд;
* интеграционных сценариев.

Kafka чаще используется для:

* событийных систем;
* аналитических платформ;
* потоковой обработки данных;
* Data Lakehouse.

В DLH-командах Kafka встречается значительно чаще.

### Каскадные отказы

Одной из главных проблем распределённых систем являются каскадные отказы.

Представим следующую цепочку:

```text
API Gateway
      ↓
Service A
      ↓
Service B
      ↓
Service C
```

Если Service C перестанет отвечать, проблема может распространиться вверх по всей цепочке.

В результате отказ одного компонента способен повлиять на всю систему.

Поэтому микросервисные архитектуры используют специальные механизмы защиты.

#### Timeout

Первый механизм — ограничение времени ожидания.

Плохо:

```python
response = requests.get(url)
```

Если сервис недоступен, ожидание может продолжаться очень долго.

Лучше:

```python
response = requests.get(
    url,
    timeout=5
)
```

Теперь запрос завершится через пять секунд.

Timeout является обязательной практикой для всех сетевых вызовов.

#### Retry

Иногда ошибка носит временный характер.

Например:

* кратковременный сетевой сбой;
* перегрузка сервиса;
* временная недоступность ресурса.

В таких случаях помогает повторная попытка.

Упрощённо:

```text
Request
   ↓
Error
   ↓
Retry
   ↓
Success
```

Однако бесконечные повторные попытки недопустимы.

Обычно используются:

```text
3 попытки

5 попыток

Exponential Backoff
```

#### Circuit Breaker

Если сервис стабильно недоступен, постоянные попытки обращения только ухудшают ситуацию.

Для решения этой задачи используется Circuit Breaker.

Упрощённая схема:

```text
Closed
   ↓
Failures
   ↓
Open
   ↓
Recovery
   ↓
Half Open
```

Когда количество ошибок превышает порог, Circuit Breaker открывается.

После этого новые запросы не отправляются во внешний сервис.

Система быстро возвращает ошибку и не тратит ресурсы на бесполезные попытки подключения.

#### Fallback

Иногда вместо ошибки можно вернуть упрощённый результат.

Например:

```text
Recommendations Service
      ↓
Unavailable
      ↓
Fallback
      ↓
Popular Products
```

Пользователь получает не идеальный результат, но система продолжает работать.

Такая стратегия часто используется в высоконагруженных системах.

### Docker и Kubernetes

По мере роста количества сервисов возникает новая проблема — развёртывание.

Если сервисов два или три, Docker Compose может быть достаточным.

Однако в крупных системах количество контейнеров может исчисляться сотнями.

Для управления такими системами используется Kubernetes.

Упрощённая схема:

```text
Kubernetes Cluster

├── Service A
├── Service B
├── Service C
└── Service D
```

Kubernetes отвечает за:

* запуск контейнеров;
* масштабирование;
* восстановление после сбоев;
* балансировку нагрузки;
* обновление версий.

Фактически Kubernetes становится операционной системой для контейнеров.

В большинстве современных микросервисных платформ именно Kubernetes является стандартом де-факто.

### Monitoring, Logging и Tracing

Когда система состоит из десятков сервисов, возникает вопрос наблюдаемости.

Если пользователь сообщает об ошибке, необходимо быстро понять:

* где произошла проблема;
* какой сервис её вызвал;
* какие запросы выполнялись.

Для этого используются три основных инструмента.

#### Monitoring

Monitoring отвечает на вопрос:

> Что происходит с системой?

Обычно собираются метрики:

* CPU;
* память;
* количество запросов;
* время ответа;
* количество ошибок.

Часто используются:

```text
Prometheus
Grafana
```

#### Logging

Логи отвечают на вопрос:

> Что произошло?

Например:

```text
Student created

User authenticated

Payment failed
```

Чаще всего используются:

```text
ELK Stack

OpenSearch

Loki
```

#### Tracing

Tracing отвечает на вопрос:

> Как запрос прошёл через систему?

```text
Client
   ↓
Gateway
   ↓
Service A
   ↓
Service B
   ↓
Database
```

Каждый шаг получает собственный trace-id.

Благодаря этому можно восстановить полный путь запроса.

Для tracing часто используются:

```text
Jaeger

Zipkin

OpenTelemetry
```

Именно сочетание Metrics, Logs и Traces формирует современную концепцию Observability.

### Тестирование микросервисов

По мере роста системы тестирование становится значительно сложнее.

Если в монолите достаточно проверить работу одного приложения, то в микросервисной архитектуре необходимо учитывать взаимодействие между множеством независимых компонентов.

Именно поэтому тестирование строится на нескольких уровнях.

#### Unit Tests

Unit-тесты проверяют отдельный компонент изолированно от внешних зависимостей.

Например:

```python
def test_calculate_discount():
    ...
```

или:

```python
def test_create_student():
    ...
```

Во время unit-тестирования обычно подменяются:

* база данных;
* внешние API;
* Kafka;
* файловые системы.

Главная задача — проверить бизнес-логику максимально быстро и независимо от окружения.

Обычно именно unit-тесты составляют основную часть тестовой пирамиды.

#### Integration Tests

Integration-тесты проверяют взаимодействие нескольких компонентов системы.

Например:

```text
Service
   ↓
DAO
   ↓
PostgreSQL
```

или:

```text
Service
   ↓
Kafka
```

В отличие от unit-тестов здесь уже используются реальные зависимости.

Очень часто integration-тесты запускаются через Docker Compose или Testcontainers.

Например:

```text
Application
     ↓
PostgreSQL Container
```

Главная задача — убедиться, что компоненты корректно работают вместе.

#### Contract Tests

Одной из особенностей микросервисов является большое количество API между сервисами.

Например:

```text
Orders Service
       ↓
Users Service
```

Если команда Users Service изменит контракт API, это может сломать Orders Service.

Для решения этой проблемы используются contract tests.

Контрактом считается соглашение между двумя сервисами:

* какие endpoint существуют;
* какие параметры принимаются;
* какие данные возвращаются;
* какие ошибки могут возникать.

Упрощённо:

```json
{
  "id": 1,
  "name": "Ivan"
}
```

Если сервис внезапно начнёт возвращать:

```json
{
  "user_id": 1
}
```

контракт будет нарушен.

Contract tests помогают обнаружить подобные изменения до попадания в production.

#### End-to-End Tests

E2E-тесты проверяют систему целиком.

Например:

```text
Client
   ↓
Gateway
   ↓
Orders Service
   ↓
Payments Service
   ↓
Database
```

Тест проходит через все уровни системы.

Это самый реалистичный тип тестирования.

Однако такие тесты:

* медленные;
* дорогие в поддержке;
* сложные в отладке.

Поэтому количество E2E-тестов обычно стараются ограничивать.

На практике E2E используются для проверки критически важных пользовательских сценариев.

#### Тестовая пирамида

Большинство команд придерживается следующего подхода:

```text
           E2E
            ▲
            │
      Integration
            ▲
            │
          Unit
```

Чем ниже уровень теста, тем больше таких тестов должно быть.

Unit-тесты составляют основу.

E2E-тестов обычно значительно меньше.

Такой баланс позволяет сохранять высокую скорость обратной связи.

### Нагрузочное тестирование

Даже если система работает корректно, это не означает, что она выдержит реальную нагрузку.

Например:

* 10 пользователей;
* 100 пользователей;
* 10 000 пользователей.

Поведение системы может существенно отличаться.

Нагрузочное тестирование позволяет понять:

* сколько запросов выдерживает система;
* когда начинает расти время ответа;
* какие компоненты становятся узким местом;
* где появляется деградация производительности.

Упрощённая схема выглядит следующим образом:

```text
Users
   ↓
Load Generator
   ↓
System
```

Наиболее популярные инструменты:

```text
JMeter

Locust

k6

Gatling
```

В современных backend-командах всё чаще используется именно k6 благодаря простоте настройки и хорошей интеграции с CI/CD.

#### Основные метрики

Во время нагрузочного тестирования обычно анализируются:

* RPS (Requests Per Second);
* время ответа;
* процент ошибок;
* загрузка CPU;
* использование памяти.

Например:

```text
1000 RPS

95% запросов < 200 ms
```

Такие показатели позволяют оценить готовность системы к реальной эксплуатации.

### Chaos Engineering

Большинство тестов предполагает, что система работает корректно.

Однако в реальной жизни постоянно происходят сбои:

* отключаются серверы;
* пропадает сеть;
* заканчивается память;
* падают базы данных;
* недоступны внешние API.

Chaos Engineering изучает поведение системы в подобных условиях.

Главная идея:

> Сломать систему контролируемым образом и проверить её устойчивость.

Например:

```text
Service A
      ↓
Service B
```

Во время эксперимента Service B искусственно отключается.

После этого анализируется:

* восстановится ли система;
* сработает ли fallback;
* откроется ли circuit breaker;
* сохранится ли работоспособность остальных сервисов.

Один из самых известных инструментов в этой области — Chaos Monkey, созданный Netflix.

Подобные практики особенно важны для высоконагруженных распределённых систем.

### Best Practices микросервисной архитектуры

Несмотря на разнообразие технологий, большинство успешных микросервисных платформ придерживается схожих принципов.

#### Разделение по бизнес-доменам

Сервисы должны строиться вокруг бизнес-областей, а не технических слоёв.

Хорошо:

```text
Users Service

Orders Service

Payments Service
```

Плохо:

```text
Database Service

Validation Service

Controllers Service
```

Первый подход лучше отражает структуру бизнеса.

#### Независимое развёртывание

Каждый сервис должен иметь возможность выпускать новые версии независимо от остальных.

Если для обновления одного сервиса необходимо пересобирать всю систему, это уже сигнал о чрезмерной связанности.

#### Собственная база данных

Каждый сервис должен владеть своими данными.

Это один из ключевых принципов микросервисной архитектуры.

#### Контракты важнее реализации

Другие сервисы не должны знать внутреннее устройство приложения.

Они должны опираться только на опубликованный контракт взаимодействия.

Поэтому особую роль играют:

* OpenAPI;
* Contract Tests;
* Schema Registry.

#### Автоматизация

Микросервисы практически невозможно поддерживать вручную.

Поэтому обязательными становятся:

* CI/CD;
* автоматизированное тестирование;
* мониторинг;
* централизованное логирование.

### Как микросервисы связаны с Data Platform

На этом этапе становится понятно, почему backend, pytest и CI/CD изучались раньше.

Микросервисы являются источником данных для аналитической платформы.

Представим создание нового студента.

Сначала происходит HTTP-запрос:

```text
Client
   ↓
POST /students
```

После этого сервис сохраняет данные в PostgreSQL.

Затем публикует событие:

```text
StudentCreated
```

Событие попадает в Kafka.

Дальше данные начинают жить собственной жизнью.

Они могут:

* попасть в Data Lake;
* использоваться аналитиками;
* участвовать в ML-моделях;
* отображаться в BI-системах.

Упрощённо путь выглядит следующим образом:

```text
Backend Service
       ↓
Kafka
       ↓
Data Lake
       ↓
Spark
       ↓
Analytics
```

Именно этот путь данных станет основной темой следующей главы, посвящённой S3, Spark, Iceberg и Data Lakehouse.

Микросервисы не только обрабатывают запросы, но и постоянно создают данные: события, изменения статусов, платежи, профили, логи. Эти данные становятся сырьём для аналитической платформы, поэтому следующая тема — Data Platform и Data Lakehouse.

## Data Platform и Data Lakehouse

> **Краткий ответ.** Data Lakehouse — это подход к хранению и обработке данных, который объединяет гибкость Data Lake и управляемость Data Warehouse. Данные обычно попадают в S3 или совместимое объектное хранилище, хранятся в форматах вроде Parquet, обрабатываются Spark и управляются табличным форматом вроде Apache Iceberg. В DLH-команде важно понимать путь данных от источника до Bronze, Silver, Gold-слоёв и аналитики.

### Эволюция хранения данных

По мере роста компании количество данных начинает быстро увеличиваться.

Сначала данные используются только самим приложением.

Например:

```text
FastAPI
   ↓
PostgreSQL
```

Такой подход хорошо работает для операционных задач.

Однако со временем появляются новые потребности:

* аналитика;
* отчёты;
* прогнозирование;
* машинное обучение;
* финансовые расчёты.

Для подобных сценариев обычной базы данных уже недостаточно.

Именно поэтому появились специализированные аналитические платформы.

### Data Warehouse

Data Warehouse (DWH) — это централизованное аналитическое хранилище данных.

Главная идея заключается в том, что данные предварительно очищаются, преобразуются и приводятся к единому виду.

После этого аналитики работают уже с подготовленной информацией.

Упрощённая схема:

```text
Sources
    ↓
ETL
    ↓
Data Warehouse
    ↓
BI
```

Преимущества DWH:

* высокая производительность аналитических запросов;
* единая структура данных;
* удобство построения отчётов.

Недостаток заключается в том, что перед загрузкой данные необходимо заранее подготовить.

### Data Lake

Следующим этапом развития стал Data Lake.

Вместо хранения только подготовленных данных система начинает хранить практически всё.

Например:

* JSON;
* CSV;
* Parquet;
* логи;
* события Kafka;
* изображения;
* файлы.

Схема выглядит следующим образом:

```text
Sources
    ↓
Data Lake
```

Данные сохраняются практически без изменений.

Преимущество такого подхода — гибкость.

Недостаток — отсутствие единой структуры и сложность управления данными.

Со временем многие Data Lake превращались в так называемое Data Swamp — большое хранилище данных, в котором трудно разобраться.

### Data Lakehouse

Data Lakehouse объединяет преимущества двух предыдущих подходов.

С одной стороны сохраняется гибкость Data Lake.

С другой стороны появляются механизмы управления данными, характерные для Data Warehouse.

Упрощённая схема:

```text
Sources
    ↓
Data Lakehouse
    ↓
Analytics
```

Сегодня именно Lakehouse считается наиболее популярной архитектурой для Data Platform.

Ключевую роль в таких системах играют:

* S3;
* Spark;
* Iceberg;
* Delta Lake;
* Hive Metastore;
* Trino;
* BI-системы.

В DLH-командах именно эти технологии используются чаще всего.

### ETL и ELT

Практически любые данные проходят через процессы преобразования.

Исторически использовалась схема ETL.

#### ETL

ETL расшифровывается как:

```text
Extract
   ↓
Transform
   ↓
Load
```

Сначала данные извлекаются.

Затем преобразуются.

После этого загружаются в хранилище.

Например:

```text
PostgreSQL
      ↓
Transformation
      ↓
Data Warehouse
```

Этот подход долгое время был стандартом аналитических систем.

#### ELT

С развитием облачных платформ всё чаще используется ELT.

```text
Extract
   ↓
Load
   ↓
Transform
```

Сначала данные загружаются в хранилище.

Трансформация выполняется уже внутри платформы.

Например:

```text
PostgreSQL
      ↓
S3
      ↓
Spark
```

Именно ELT сегодня является наиболее распространённым подходом в Lakehouse-платформах.

### Bronze, Silver и Gold

В большинстве современных Data Platform используется многоуровневая модель хранения данных.

Наиболее популярный вариант — архитектура Bronze, Silver и Gold.

#### Bronze Layer

Bronze содержит сырые данные.

Например:

```text
Kafka Events

API Logs

Raw JSON

CSV Files
```

На этом уровне данные практически не изменяются.

Главная задача — сохранить информацию в первоначальном виде.

#### Silver Layer

На уровне Silver выполняется очистка и нормализация данных.

Например:

* удаление дубликатов;
* исправление форматов;
* приведение схемы к стандартному виду.

Получаются качественные данные, пригодные для дальнейшей обработки.

#### Gold Layer

На уровне Gold появляются бизнес-модели.

Например:

```text
Daily Revenue

Active Users

Student Statistics
```

Именно Gold обычно используется аналитиками и BI-инструментами.

Полный путь выглядит следующим образом:

```text
Bronze
   ↓
Silver
   ↓
Gold
```

Такой подход позволяет чётко разделять этапы жизненного цикла данных.

### Как выглядит DLH в реальной системе

Представим backend-сервис студентов.

Пользователь создаёт нового студента:

```http
POST /students
```

Сервис сохраняет запись в PostgreSQL.

После этого публикует событие:

```text
StudentCreated
```

Далее начинается путь данных внутри платформы:

```text
PostgreSQL
     ↓
Kafka
     ↓
S3 Bronze
     ↓
Spark
     ↓
S3 Silver
     ↓
Spark
     ↓
Iceberg Gold
     ↓
BI
```

Фактически DLH становится продолжением backend-системы.

Если backend отвечает за создание данных, то Data Platform отвечает за их дальнейшую жизнь и использование.

### S3

В большинстве современных Data Lakehouse-платформ данные физически хранятся в объектном хранилище.

Наиболее популярным вариантом является Amazon S3 или совместимые решения:

* MinIO;
* Ceph;
* Yandex Object Storage;
* VK Cloud Object Storage.

Несмотря на название, S3 давно перестал быть исключительно AWS-сервисом. Сегодня S3 API фактически стал стандартом объектного хранения данных.

Для Data Engineering и DLH S3 играет ту же роль, что PostgreSQL для backend-приложений.

Если backend хранит операционные данные в базе данных, то аналитическая платформа хранит большие объёмы данных в S3.

#### Основные сущности S3

S3 хранит данные в виде объектов.

В отличие от файловой системы здесь нет настоящих директорий и файлов.

Существуют только объекты и их ключи.

#### Bucket

Bucket можно представить как логический контейнер для хранения данных.

Например:

```text
students-datalake

analytics

raw-events
```

Все объекты всегда принадлежат конкретному bucket.

Упрощённо:

```text
Bucket
   ↓
Objects
```

Название bucket должно быть уникальным в пределах конкретного S3-хранилища.

#### Object

Object является основной единицей хранения данных.

Например:

```text
students.parquet

events.json

report.csv
```

Объект состоит из:

* данных;
* метаданных;
* ключа.

Фактически object является аналогом файла.

#### Key

Каждый объект имеет уникальный key.

Например:

```text
bronze/students/2025/01/data.parquet
```

Важно понимать, что это не путь к файлу.

Для S3 это просто строка.

Именно key используется для поиска объекта внутри bucket.

#### Prefix

Хотя директорий в S3 не существует, ключи часто организуются так, будто они есть.

Например:

```text
bronze/students/
bronze/orders/
silver/students/
gold/reports/
```

Такие части ключа называют prefix.

Prefix позволяет логически группировать данные.

Практически все DLH-платформы активно используют этот механизм.

#### Metadata

Каждый объект может содержать дополнительную информацию.

Например:

```text
Content-Type

Owner

Created-By

Source-System
```

Метаданные помогают управлять данными и используются многими инструментами Data Platform.

### Работа с S3 через boto3

Основной Python SDK для работы с S3 называется boto3.

Создание клиента:

```python
import boto3

s3 = boto3.client("s3")
```

После этого можно выполнять операции с объектами.

#### Загрузка файла

```python
s3.upload_file(
    "students.parquet",
    "analytics-bucket",
    "bronze/students.parquet"
)
```

Файл будет загружен в bucket.

#### Скачивание файла

```python
s3.download_file(
    "analytics-bucket",
    "bronze/students.parquet",
    "students.parquet"
)
```

Объект будет сохранён локально.

#### Получение списка объектов

```python
response = s3.list_objects_v2(
    Bucket="analytics-bucket"
)
```

После этого можно получить список найденных объектов.

```python
for obj in response["Contents"]:
    print(obj["Key"])
```

#### Удаление объекта

```python
s3.delete_object(
    Bucket="analytics-bucket",
    Key="bronze/students.parquet"
)
```

После выполнения объект будет удалён.

### Работа через Resource API

Помимо client существует более высокоуровневый интерфейс.

```python
import boto3

s3 = boto3.resource("s3")
```

Получение объекта:

```python
bucket = s3.Bucket(
    "analytics-bucket"
)
```

Получение списка файлов:

```python
for obj in bucket.objects.all():
    print(obj.key)
```

На практике в Data Engineering чаще используется именно client API, поскольку он предоставляет полный набор возможностей S3.

### s3fs

Очень часто данные необходимо читать напрямую в Pandas или Spark.

Для этого удобно использовать библиотеку s3fs.

Например:

```python
import pandas as pd

df = pd.read_parquet(
    "s3://analytics-bucket/students.parquet"
)
```

Без s3fs такой код работать не будет.

Библиотека позволяет обращаться к объектам S3 как к обычной файловой системе.

Именно поэтому она широко используется в аналитических пайплайнах.

### aioboto3

Если приложение работает асинхронно, обычный boto3 может стать узким местом.

В таких случаях используется aioboto3.

Пример:

```python
import aioboto3

async with aioboto3.Session().client(
    "s3"
) as s3:

    response = await s3.list_buckets()
```

Подобный подход часто встречается в FastAPI-сервисах и асинхронных ETL-процессах.

### Paginator

Одна из частых ошибок новичков связана с получением списка объектов.

Представим bucket с миллионами файлов.

Запрос:

```python
s3.list_objects_v2(...)
```

не сможет вернуть всё сразу.

S3 разбивает результат на страницы.

Для работы с большими объёмами данных используется paginator.

Пример:

```python
paginator = s3.get_paginator(
    "list_objects_v2"
)

pages = paginator.paginate(
    Bucket="analytics-bucket"
)
```

После этого можно безопасно обходить все страницы.

```python
for page in pages:

    for obj in page.get(
        "Contents",
        []
    ):
        print(obj["Key"])
```

В production-коде при работе с большими bucket paginator считается обязательной практикой.

### Presigned URL

Иногда необходимо предоставить доступ к объекту без выдачи AWS-ключей.

Для этого используются Presigned URL.

Сервис генерирует временную ссылку:

```python
url = s3.generate_presigned_url(
    "get_object",
    Params={
        "Bucket": "analytics",
        "Key": "report.pdf"
    },
    ExpiresIn=3600
)
```

После этого пользователь может скачать файл по ссылке.

При этом прямого доступа к S3 у него нет.

Очень часто Presigned URL используются:

* для загрузки файлов из браузера;
* для скачивания отчётов;
* для обмена большими файлами;
* для интеграций между сервисами.

### Организация данных в DLH

На практике структура S3 обычно отражает слои Lakehouse.

Например:

```text
s3://datalake/

├── bronze/
│
├── silver/
│
└── gold/
```

Внутри появляются отдельные домены данных:

```text
bronze/

├── students/
├── orders/
├── payments/
```

или:

```text
silver/

├── students/
├── orders/
└── users/
```

Подобная организация позволяет Spark и Iceberg эффективно работать с данными.

Кроме того, такая структура делает хранилище понятным для инженеров и аналитиков.

### Роль S3 в Data Platform

Если посмотреть на полный путь данных, то становится понятно, почему S3 является фундаментом большинства DLH-платформ.

Упрощённо поток выглядит следующим образом:

```text
Backend Service
       ↓
Kafka
       ↓
S3 Bronze
       ↓
Spark
       ↓
S3 Silver
       ↓
Spark
       ↓
Iceberg Gold
```

S3 становится центральным хранилищем, через которое проходят практически все данные платформы.

Именно поэтому понимание bucket, object, prefix, boto3 и организации данных в S3 считается базовым навыком для Data Engineer и DLH-разработчика.

### Форматы данных

Прежде чем перейти к Spark, важно разобраться в форматах хранения данных.

От выбранного формата напрямую зависит:

* скорость чтения;
* объём занимаемого места;
* производительность аналитических запросов;
* стоимость хранения данных.

В современных Data Lakehouse-платформах чаще всего используются четыре формата:

* CSV;
* JSON;
* Parquet;
* ORC.

#### CSV

CSV (Comma Separated Values) — один из самых старых и простых форматов хранения данных.

Пример:

```csv
id,name,age
1,Ivan,20
2,Anna,21
```

Преимущества:

* простота;
* читаемость человеком;
* поддержка практически всеми системами.

Недостатки:

* отсутствие информации о типах данных;
* низкая эффективность хранения;
* плохая производительность на больших объёмах.

CSV хорошо подходит для обмена данными, но редко используется как основной формат аналитического хранения.

#### JSON

JSON широко применяется в backend-разработке и API.

Пример:

```json
{
  "id": 1,
  "name": "Ivan",
  "age": 20
}
```

Также часто используется формат JSON Lines:

```json
{"id":1,"name":"Ivan"}
{"id":2,"name":"Anna"}
```

Преимущества:

* удобство интеграции;
* поддержка вложенных структур;
* хорошая совместимость с API.

Недостатки:

* большой объём данных;
* медленная аналитическая обработка;
* отсутствие эффективного сжатия.

В DLH JSON чаще встречается на Bronze-слое.

#### Parquet

Parquet является стандартом де-факто для современных Lakehouse-платформ.

Главное отличие заключается в колонночном хранении данных.

Упрощённо:

Обычная строковая структура:

```text
id name age
id name age
id name age
```

Parquet:

```text
id
id
id

name
name
name

age
age
age
```

Благодаря этому аналитические системы могут читать только нужные колонки.

Например:

```sql
SELECT age
FROM students
```

В этом случае Spark не будет читать столбцы name и id.

Преимущества:

* высокая скорость чтения;
* эффективное сжатие;
* поддержка схем;
* оптимизация аналитических запросов.

Практически все современные DLH-платформы используют именно Parquet как базовый формат хранения данных.

#### ORC

ORC (Optimized Row Columnar) решает похожие задачи.

Он также является колонночным форматом.

Исторически ORC получил распространение в экосистеме Hadoop и Hive.

Сегодня чаще встречаются:

* Parquet;
* Iceberg + Parquet.

Поэтому в большинстве современных проектов именно Parquet становится основным форматом хранения.

### Spark и PySpark

Когда объём данных начинает измеряться гигабайтами и терабайтами, обработка через Pandas становится проблематичной.

Pandas прекрасно работает на одном компьютере.

Однако для действительно больших объёмов данных используются распределённые вычисления.

Именно эту задачу решает Apache Spark.

#### Что такое Spark

Apache Spark — распределённый движок обработки данных.

Главная идея заключается в том, что вычисления могут выполняться одновременно на нескольких узлах кластера.

Упрощённо:

```text
Spark Application
         ↓
      Cluster
         ↓
 ┌───────┬───────┬───────┐
 │       │       │
Node1  Node2   Node3
```

Каждый узел выполняет часть общей работы.

Благодаря этому Spark способен обрабатывать огромные объёмы данных.

#### Почему Spark используется в DLH

Практически все современные Data Lakehouse-платформы используют Spark для:

* ETL-процессов;
* очистки данных;
* агрегаций;
* построения витрин;
* записи данных в Iceberg.

Упрощённо:

```text
S3
 ↓
Spark
 ↓
Iceberg
 ↓
Analytics
```

Поэтому Spark является одной из ключевых технологий Data Engineering.

### PySpark

PySpark представляет собой Python API для Apache Spark.

Вместо Scala или Java разработчик может использовать привычный Python.

Пример создания Spark-приложения:

```python
from pyspark.sql import SparkSession

spark = SparkSession.builder.getOrCreate()
```

Практически любое Spark-приложение начинается именно с создания SparkSession.

### SparkSession

SparkSession является точкой входа в Spark.

Через него выполняются:

* чтение данных;
* создание DataFrame;
* запись результатов;
* работа с каталогами.

Пример чтения файла:

```python
df = spark.read.parquet(
    "s3://datalake/students"
)
```

После этого данные становятся доступны для обработки.

Без SparkSession работа со Spark невозможна.

### DataFrame

Основной структурой данных в Spark является DataFrame.

Концептуально он похож на DataFrame из Pandas.

Например:

```python
df.show()
```

выведет содержимое таблицы.

Получение схемы:

```python
df.printSchema()
```

Получение колонок:

```python
df.columns
```

Однако существует важное отличие.

Spark DataFrame является распределённым.

Данные могут находиться сразу на нескольких узлах кластера.

Поэтому операции выполняются иначе, чем в Pandas.

### Schema

Schema определяет структуру данных.

Например:

```text
id: integer
name: string
age: integer
```

Spark может определять схему автоматически.

Однако в production-проектах предпочтительно задавать её явно.

Пример:

```python
from pyspark.sql.types import *

schema = StructType([
    StructField(
        "id",
        IntegerType()
    ),
    StructField(
        "name",
        StringType()
    )
])
```

Явная схема:

* уменьшает количество ошибок;
* ускоряет чтение данных;
* делает код более предсказуемым.

### Lazy Execution

Одна из важнейших концепций Spark.

Большинство операций не выполняется сразу.

Например:

```python
df = spark.read.parquet(...)

df = df.filter(
    df.age > 18
)

df = df.select(
    "name",
    "age"
)
```

Несмотря на три операции, Spark пока ничего не выполняет.

Он лишь строит план вычислений.

Реальное выполнение начнётся только тогда, когда потребуется результат.

Именно это называется Lazy Execution.

Spark сначала строит DAG (Directed Acyclic Graph), а затем оптимизирует выполнение.

Благодаря этому достигается высокая производительность.

### Transformations

Операции, которые создают новый DataFrame, называются transformations.

Например:

```python
filter()
select()
groupBy()
join()
withColumn()
drop()
```

Пример:

```python
df_filtered = df.filter(
    df.age > 18
)
```

Новый DataFrame будет создан логически, но вычисления пока не начнутся.

Transformations являются ленивыми операциями.

### Actions

Actions запускают выполнение вычислений.

Например:

```python
show()

count()

collect()

write()
```

Пример:

```python
df.count()
```

После вызова count Spark начнёт выполнять построенный ранее план вычислений.

Именно actions переводят Spark из состояния планирования в состояние выполнения.

Это один из самых популярных вопросов на собеседованиях по Spark.

### Partitioning

Для эффективной работы большие наборы данных разбиваются на partition.

Упрощённо:

```text
students

├── part-1
├── part-2
├── part-3
└── part-4
```

Каждый partition может обрабатываться независимо.

Это позволяет использовать параллелизм.

В DLH partitioning играет критически важную роль.

Например:

```text
students/

year=2025/
month=01/

year=2025/
month=02/
```

Теперь Spark может читать только необходимые данные.

Если аналитик выполняет запрос:

```sql
WHERE year = 2025
```

Spark сможет пропустить ненужные partition.

Это существенно уменьшает объём чтения данных и ускоряет выполнение запросов.

### Как Spark выглядит внутри DLH

На практике Spark обычно располагается между слоями хранения данных.

Например:

```text
Bronze
   ↓
Spark ETL
   ↓
Silver
   ↓
Spark ETL
   ↓
Gold
```

Или в более полном виде:

```text
Kafka
   ↓
S3 Bronze
   ↓
Spark
   ↓
S3 Silver
   ↓
Spark
   ↓
Iceberg Gold
```

Именно Spark выполняет основную работу по преобразованию данных между слоями платформы.

Поэтому понимание DataFrame, Lazy Execution, Transformations и Partitioning считается обязательным для любого Data Engineer или DLH-разработчика.

### Apache Iceberg

До появления Iceberg большинство Data Lake строилось на обычных Parquet-файлах.

На первый взгляд такой подход кажется удобным:

```text
S3
 ↓
Parquet Files
```

Однако по мере роста объёма данных начинают возникать серьёзные проблемы.

Например:

* как гарантировать консистентность данных;
* как отслеживать версии таблиц;
* как выполнять rollback;
* как изменять схему данных;
* как обновлять только часть данных;
* как организовать ACID-операции.

Обычные файлы Parquet не умеют решать эти задачи самостоятельно.

Именно поэтому появились форматы таблиц Data Lakehouse:

* Apache Iceberg;
* Delta Lake;
* Apache Hudi.

В большинстве современных DLH-платформ всё чаще используется именно Apache Iceberg.

Главная идея Iceberg заключается в том, что он управляет не данными напрямую, а метаданными таблицы.

Физические файлы по-прежнему лежат в S3.

Но поверх них появляется дополнительный уровень управления.

Упрощённо:

```text
Iceberg Table
      ↓
Metadata
      ↓
Parquet Files
      ↓
S3
```

Именно этот слой метаданных делает Lakehouse похожим на полноценную базу данных.

### Основные компоненты Iceberg

Внутри Iceberg существует несколько типов файлов.

Они работают совместно и обеспечивают управление таблицей.

#### Data Files

Data Files содержат сами данные.

Обычно это обычные Parquet-файлы.

Например:

```text
data/

part-0001.parquet
part-0002.parquet
part-0003.parquet
```

Именно здесь находятся строки таблицы.

Для Spark это обычные файлы данных.

#### Metadata Files

Metadata Files являются сердцем Iceberg.

Они описывают состояние таблицы.

Например:

* текущую схему;
* список snapshot;
* информацию о partition;
* ссылки на manifest-файлы.

Упрощённо:

```text
metadata.json
      ↓
manifests
      ↓
data files
```

Когда Spark открывает таблицу Iceberg, сначала читаются именно metadata-файлы.

#### Manifest Files

Manifest содержит информацию о группах data files.

Например:

```text
Manifest A

 ├── file_1.parquet
 ├── file_2.parquet
 └── file_3.parquet
```

Благодаря manifest Spark не обязан сканировать весь S3.

Он быстро понимает, какие файлы относятся к нужной таблице.

Это существенно ускоряет выполнение запросов.

#### Snapshots

Snapshot представляет собой версию таблицы в конкретный момент времени.

Например:

```text
Snapshot 1

Snapshot 2

Snapshot 3
```

Каждое изменение создаёт новый snapshot.

При этом старые версии сохраняются.

Именно snapshots позволяют реализовать многие возможности Iceberg.

Например:

* rollback;
* audit;
* time travel;
* безопасные обновления.

Фактически snapshot можно воспринимать как "снимок состояния таблицы".

### Catalog

Возникает вопрос:

> Как Spark узнаёт о существовании таблицы?

Для этого используется Catalog.

Catalog хранит информацию о таблицах и их метаданных.

Упрощённо:

```text
Catalog
    ↓
Table
    ↓
Metadata
```

На практике используются разные варианты:

```text
Hive Metastore

AWS Glue

 Nessie

 JDBC Catalog
```

Именно через Catalog Spark получает доступ к Iceberg-таблицам.

Например:

```python
spark.table(
    "students"
)
```

В этот момент Spark обращается к Catalog и получает информацию о таблице.

### Time Travel

Одна из самых известных возможностей Iceberg.

Поскольку каждая версия таблицы сохраняется в виде snapshot, можно читать данные из прошлого.

Например:

```text
Snapshot 100
      ↓
Snapshot 101
      ↓
Snapshot 102
```

Можно запросить состояние таблицы на момент Snapshot 100.

Пример:

```sql
SELECT *
FROM students
VERSION AS OF 100
```

или:

```sql
SELECT *
FROM students
TIMESTAMP AS OF '2025-01-01'
```

Подобная функциональность чрезвычайно полезна для:

* расследования ошибок;
* восстановления данных;
* аудита изменений;
* аналитики.

В обычном наборе Parquet-файлов подобное реализовать значительно сложнее.

### Schema Evolution

Со временем структура данных меняется.

Например, первоначально таблица выглядела так:

```text
id
name
age
```

Позже появляется новое поле:

```text
email
```

В традиционных Data Lake такие изменения часто приводили к проблемам совместимости.

Iceberg поддерживает Schema Evolution.

Например:

```sql
ALTER TABLE students
ADD COLUMN email STRING
```

После этого новая схема начинает использоваться автоматически.

При этом старые данные продолжают оставаться доступными.

Поддерживаются операции:

* добавление колонок;
* удаление колонок;
* переименование колонок;
* изменение порядка колонок.

Это значительно упрощает сопровождение аналитических систем.

### Partition Evolution

Ещё одна проблема больших таблиц связана с partitioning.

Представим, что изначально таблица была разбита по дням.

```text
year
month
day
```

Со временем объём данных вырос.

Появилась необходимость использовать другой способ разбиения.

Например:

```text
year
month
```

В большинстве традиционных систем это крайне болезненная операция.

Iceberg поддерживает Partition Evolution.

Новая схема partition может использоваться без полной перезаписи таблицы.

Это особенно важно для таблиц объёмом в десятки и сотни терабайт.

### Hidden Partitioning

Одна из наиболее удобных возможностей Iceberg.

В классических Hive-таблицах разработчик обязан знать структуру partition.

Например:

```sql
WHERE year = 2025
```

В Iceberg детали partition скрываются от пользователя.

Разработчик пишет обычный запрос:

```sql
SELECT *
FROM students
WHERE created_at >= '2025-01-01'
```

А Iceberg самостоятельно определяет нужные partition.

Поэтому механизм называется Hidden Partitioning.

Он упрощает разработку и уменьшает количество ошибок.

### PyIceberg

PyIceberg предоставляет Python API для работы с Iceberg.

Это особенно удобно для:

* ETL-процессов;
* сервисных задач;
* автоматизации;
* проверки метаданных.

Пример загрузки таблицы:

```python
from pyiceberg.catalog import load_catalog

catalog = load_catalog("default")

table = catalog.load_table(
    "students"
)
```

Получение схемы:

```python
print(
    table.schema()
)
```

Получение snapshot:

```python
table.current_snapshot()
```

PyIceberg становится всё более популярным инструментом в Data Engineering, особенно в Python-ориентированных командах.

### Как Iceberg работает поверх S3

Очень важно понимать, что Iceberg не заменяет S3.

Он работает поверх него.

Упрощённо структура выглядит следующим образом:

```text
S3

├── data/
│   ├── part-1.parquet
│   ├── part-2.parquet
│   └── part-3.parquet
│
└── metadata/
    ├── metadata.json
    ├── manifest-1.avro
    └── manifest-2.avro
```

Данные продолжают храниться в объектном хранилище.

Iceberg лишь управляет ими через систему метаданных.

Поэтому часто можно услышать формулировку:

> Iceberg — это table format для Data Lake.

Это достаточно точное описание.

### Полный путь данных в DLH

Теперь можно собрать все изученные технологии в единую цепочку.

Представим создание нового студента через backend-сервис.

Сначала пользователь отправляет запрос:

```http
POST /students
```

FastAPI сохраняет данные в PostgreSQL.

После этого сервис публикует событие:

```text
StudentCreated
```

Далее начинается работа Data Platform.

```text
Backend Service
       ↓
PostgreSQL
       ↓
Kafka
       ↓
Bronze (S3)
       ↓
Spark
       ↓
Silver (S3)
       ↓
Spark
       ↓
Iceberg Gold
       ↓
BI / Analytics
```

На уровне Bronze сохраняются сырые данные.

На уровне Silver выполняется очистка и нормализация.

На уровне Gold появляются бизнес-модели и витрины данных.

Spark выполняет преобразования.

S3 обеспечивает хранение.

Iceberg управляет таблицами и метаданными.

BI-системы используют подготовленные данные для аналитики.

Именно такая архитектура сегодня является одной из самых распространённых в современных DLH и Data Platform командах.

На этом этапе можно проследить полный жизненный цикл данных — от HTTP-запроса пользователя до аналитического отчёта, построенного поверх Iceberg-таблиц.

## Как всё связано между собой

> **Краткий ответ.** Все темы конспекта складываются в один технологический контур. Backend принимает запросы и создаёт данные, pytest проверяет код, CI/CD автоматизирует проверки и доставку, микросервисы разделяют систему на независимые компоненты, а DLH-платформа превращает события и операционные данные в аналитический слой. Поэтому важно видеть не только отдельные инструменты, но и полный путь данных внутри компании.

К этому моменту были рассмотрены backend-разработка, тестирование, CI/CD, микросервисы и Data Lakehouse. На первый взгляд эти темы могут казаться отдельными направлениями, однако в реальной системе они образуют единый технологический контур.

Современное приложение не заканчивается на API и базе данных.

Данные, созданные внутри backend-сервисов, используются другими сервисами, попадают в аналитические платформы, участвуют в отчётах, машинном обучении и бизнес-аналитике.

Поэтому важно понимать не только отдельные технологии, но и их место в общей архитектуре.

### Общая картина системы

Если посмотреть на путь данных сверху, получится следующая схема:

```text
Client
   ↓
FastAPI
   ↓
PostgreSQL
   ↓
Kafka
   ↓
S3 Bronze
   ↓
Spark
   ↓
S3 Silver
   ↓
Spark
   ↓
Iceberg Gold
   ↓
BI / Analytics
```

Каждая технология в этой цепочке решает собственную задачу.

Backend создаёт данные.

Kafka транспортирует события.

S3 хранит данные.

Spark выполняет обработку.

Iceberg управляет аналитическими таблицами.

BI-системы позволяют бизнесу работать с результатом.

Если убрать любой из этих компонентов, система либо перестанет работать, либо станет значительно менее удобной и масштабируемой.

### Backend как источник данных

В большинстве компаний жизненный цикл данных начинается именно с backend-сервисов.

Например:

```http
POST /students
```

Пользователь создаёт нового студента.

FastAPI принимает запрос.

Pydantic валидирует данные.

Service выполняет бизнес-логику.

DAO сохраняет данные через SQLAlchemy.

PostgreSQL хранит новую запись.

Упрощённо:

```text
HTTP Request
      ↓
FastAPI
      ↓
Service
      ↓
DAO
      ↓
PostgreSQL
```

На этом этапе задача backend считается выполненной.

Однако для аналитической платформы работа только начинается.

### Где появляется Pytest

Любое изменение backend-кода может привести к ошибке.

Например:

* сломать API;
* нарушить бизнес-логику;
* повредить интеграцию с базой данных.

Поэтому перед попаданием изменений в репозиторий запускаются тесты.

Обычно проверяются:

* сервисы;
* DAO;
* API;
* интеграции.

Упрощённо:

```text
Developer
      ↓
Pytest
      ↓
Result
```

Если тесты не проходят, изменение считается небезопасным.

Таким образом pytest становится первым механизмом контроля качества системы.

### Где появляется CI/CD

После того как разработчик отправляет изменения в репозиторий, начинается работа CI/CD.

Типичный процесс выглядит следующим образом:

```text
Git Push
    ↓
Pipeline
    ↓
Pytest
    ↓
Build
    ↓
Deploy
```

GitLab автоматически запускает проверки.

Если тесты завершились успешно, начинается сборка приложения.

После этого может выполняться деплой.

Получается следующая связь:

```text
Backend
   ↓
Pytest
   ↓
GitLab CI
```

По сути CI/CD автоматизирует запуск тех проверок, которые раньше выполнялись вручную.

### Где появляются микросервисы

По мере роста продукта один сервис начинает разделяться на несколько независимых компонентов.

Например:

```text
Students Service

Payments Service

Users Service

Notifications Service
```

Каждый сервис отвечает за собственную бизнес-область.

Каждый имеет собственную базу данных.

Каждый разворачивается независимо.

В результате появляется распределённая система.

С этого момента данные начинают активно перемещаться между сервисами.

### Почему появляется Kafka

Если каждый сервис будет напрямую вызывать остальные через HTTP, связанность системы начнёт быстро расти.

Поэтому часто используется событийная архитектура.

Например:

```text
StudentCreated
```

После создания студента сервис публикует событие.

Дальше это событие может использоваться другими компонентами.

```text
Students Service
        ↓
StudentCreated
        ↓
Kafka
        ↓
Consumers
```

В этот момент Kafka становится центральным транспортным слоем платформы.

Особенно часто такая архитектура встречается в Data Platform и DLH-командах.

### Как данные попадают в Data Lakehouse

После появления событий данные начинают поступать в аналитическую платформу.

Обычно первый слой выглядит следующим образом:

```text
Kafka
   ↓
Bronze
```

События сохраняются в S3 практически без изменений.

Например:

```json
{
  "student_id": 1,
  "name": "Ivan"
}
```

Такие данные называются сырыми.

Главная задача Bronze — сохранить информацию максимально близко к исходному виду.

### Где используется Spark

Сырые данные редко подходят для аналитики.

Например:

* встречаются дубликаты;
* отсутствуют поля;
* нарушена схема;
* используются разные форматы.

Поэтому появляются ETL или ELT-пайплайны.

Именно здесь обычно используется Spark.

Упрощённо:

```text
Bronze
   ↓
Spark
   ↓
Silver
```

Spark выполняет:

* фильтрацию;
* очистку;
* агрегацию;
* объединение данных.

После этого данные становятся значительно качественнее.

Следующий этап:

```text
Silver
   ↓
Spark
   ↓
Gold
```

На уровне Gold появляются бизнес-витрины и аналитические модели.

### Зачем нужен Iceberg

Когда объём данных начинает измеряться терабайтами, простого хранения Parquet-файлов становится недостаточно.

Появляются требования:

* версионирование;
* rollback;
* schema evolution;
* time travel;
* управление метаданными.

Для решения этих задач используется Apache Iceberg.

Фактически Iceberg становится аналогом базы данных поверх S3.

Упрощённо:

```text
S3
 ↓
Parquet
 ↓
Iceberg
```

При этом данные продолжают храниться в объектном хранилище.

Iceberg отвечает за управление таблицами и их состоянием.

### Где появляется аналитика

После обработки данных ими начинают пользоваться другие команды.

Например:

* аналитики;
* продуктовые менеджеры;
* data scientists;
* ML-инженеры.

Обычно используются инструменты визуализации:

```text
Power BI

Superset

Tableau

Metabase
```

Именно на этом этапе данные превращаются в бизнес-ценность.

Пользователь создал студента через API, а спустя некоторое время эта запись уже участвует в аналитических отчётах компании.

### Типичный путь данных в компании

Если объединить всё вместе, получится следующая цепочка:

```text
User
  ↓
FastAPI
  ↓
PostgreSQL
  ↓
Kafka
  ↓
S3 Bronze
  ↓
Spark
  ↓
S3 Silver
  ↓
Spark
  ↓
Iceberg Gold
  ↓
BI Dashboard
```

Это одна из наиболее распространённых схем современных Data Platform.

Разумеется, в конкретной компании детали могут отличаться.

Однако общая логика обычно остаётся именно такой.

### Роль QA Backend

Backend QA работает на пересечении разработки и тестирования.

Основные задачи:

* проверка API;
* написание автотестов;
* тестирование интеграций;
* проверка контрактов;
* участие в CI/CD.

Такой специалист хорошо понимает устройство backend-системы и помогает поддерживать качество продукта.

Именно с этой роли многие инженеры начинают переход в разработку.

### Роль Backend Developer

Backend-разработчик отвечает за создание сервисов и бизнес-логики.

Типичные задачи:

* разработка API;
* работа с базами данных;
* интеграции между сервисами;
* проектирование архитектуры;
* работа с Kafka;
* поддержка CI/CD.

Фактически backend является источником большинства данных внутри компании.

### Роль Data Engineer / DLH Engineer

Data Engineer отвечает за платформу обработки данных.

Типичные задачи:

* разработка ETL и ELT;
* работа с Spark;
* работа с S3;
* разработка Iceberg-таблиц;
* поддержка Data Lakehouse;
* обеспечение качества данных;
* оптимизация производительности пайплайнов.

Если backend отвечает за создание данных, то Data Engineer отвечает за их дальнейшую жизнь внутри платформы.

### Краткая шпаргалка перед собеседованием

#### Backend: суть

* API — интерфейс взаимодействия между системами.
* HTTP состоит из запросов и ответов.
* FastAPI использует Pydantic для валидации и Depends для Dependency Injection.
* SQLAlchemy отвечает за работу с БД.
* DAO изолирует слой доступа к данным.
* Alembic используется для миграций.
* Docker обеспечивает воспроизводимое окружение.

#### Pytest: суть

* Основной инструмент тестирования Python.
* Fixture создаёт тестовое окружение.
* Yield используется для setup и teardown.
* Scope определяет время жизни фикстуры.
* Parametrize позволяет запускать тест с разными данными.
* Хуки изменяют поведение pytest.
* Плагины расширяют возможности фреймворка.

#### CI/CD: суть

* CI автоматически проверяет изменения.
* CD автоматически доставляет изменения.
* Pipeline состоит из stages и jobs.
* GitLab CI описывается через `.gitlab-ci.yml`.
* Artifacts передают файлы между job.
* Needs ускоряет выполнение pipeline.
* Rules управляет запуском job.

#### Микросервисы: суть

* Каждый сервис отвечает за одну бизнес-область.
* Каждый сервис владеет собственной БД.
* API Gateway является единой точкой входа.
* Kafka используется для событийного взаимодействия.
* Circuit Breaker защищает систему от каскадных отказов.
* Monitoring, Logging и Tracing обеспечивают наблюдаемость.

#### DLH: суть

* Data Lake хранит данные.
* Data Warehouse хранит подготовленные данные.
* Data Lakehouse объединяет оба подхода.
* Bronze → Silver → Gold описывает жизненный цикл данных.
* S3 используется как основное хранилище.
* Spark выполняет обработку данных.
* Iceberg управляет таблицами поверх S3.
* Time Travel позволяет читать исторические версии данных.
* Schema Evolution позволяет безопасно менять схему таблиц.

Главная мысль всего конспекта заключается в том, что современные backend- и data-платформы представляют собой единую систему. Данные рождаются в API, проходят через микросервисы и очереди сообщений, сохраняются в Data Lakehouse, обрабатываются Spark и становятся основой для аналитики и принятия бизнес-решений.
